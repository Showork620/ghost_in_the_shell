# EC2 インスタンス購入オプション（料金モデル）

| 購入オプション | 割引率（目安） | 柔軟性・特徴 | 主なユースケース |
| :--- | :--- | :--- | :--- |
| **オンデマンド**<br>（デフォルト） | 基準料金 | 利用時間に対する従量課金 | 開発環境、一時的なシステム |
| **スタンダードRI**<br>（リザーブド） | 最大72% | 1年か3年の契約。OSやインスタンスタイプの変更**不可**。<br>※AZ指定で**キャパシティ（空き容量）予約**が自動付与 | 本番環境、長期稼働が確定し、絶対に確保したいシステム |
| **コンバーティブルRI** | 最大66% | 1年か3年の契約。OSやインスタンスタイプの変更**可能**。 | 本番環境、変更があるかもしれない長期稼働システム |
| **Savings Plans (SP)**<br>▶ Compute | 最大66% | **1年か3年の契約**。**【最も柔軟】** 1時間あたりの利用金額を約束。<br>ファミリー、サイズ、OS、リージョンの変更自由。 | 近代的な柔軟なシステム開発・運用全般（推奨） |
| **Savings Plans (SP)**<br>▶ EC2 Instance | 最大72% | **1年か3年の契約**。特定のファミリーとリージョンに限定されるが、割引率が高い。 | ある程度構成が固まっているシステムのコスト最適化 |
| *(SPの注意点)* | - | RIの弱点を克服した後発プラン。※キャパシティ予約機能は無いため、必要な場合は別途「オンデマンドキャパシティ予約」と併用する。 | - |
| **スポット** | 最大90% | 各AZの予備領域を安く利用。利用中に**中断の可能性**あり。 | バッチ処理など、途中で止まっても問題ない処理 |

# 物理サーバーの専有モデル

| オプション | 物理サーバーの可視性 | EC2配置の制御 | 特徴・ユースケース |
| :--- | :---: | :---: | :--- |
| **ハードウェア専有**<br>（Dedicated Instance） | × | ×（AWS任せ） | 他のAWSアカウントと物理的なサーバーを共有しない。<br>マルチテナント回避という**コンプライアンス要件**を満たすためだけに使用。 |
| **専有ホスト**<br>（Dedicated Host） | 〇 | 〇（細かく制御可） | 物理サーバーを丸ごと1台割り当て。ハードウェア情報（コア数など）が見える。<br>【試験重要】既存のOSやDBのライセンスを持ち込む**BYOL (Bring Your Own License)**に必須。 |

# インスタンスファミリー

| ファミリー | 略語の意味 | 分類 | 用途 |
| :---: | :--- | :--- | :--- |
| **M** | **M**ultipurpose | 汎用 | リソースのバランスが取れたファミリー。特定リソースに偏りのない多くの環境に適している。 |
| **T** | **T**urbo（バースト） | 汎用（CPUバースト機能） | バランス型だが、アイドル時間が多い小規模サーバーや開発環境向け。<br>一時的に制限以上のCPU性能を発揮する**バースト機能**あり。 |
| **C** | **C**ompute | コンピューティング最適化 | 高性能CPUを提供。負荷の高いバッチ処理やアクセス頻度の高いWebサーバーなど。 |
| **R、X** | **R**AM / E**x**tra large memory | メモリ最適化 | 大容量メモリを提供。リアルタイムのビッグデータ解析、アクセス頻度の高いDBサーバーなど。 |
| **P、G、F** | **P**icture / **G**raphics / **F**PGA | 高速コンピューティング | 高性能GPUを提供。機械学習や3Dグラフィックス処理など、高速演算が必要な環境向け。 |
| **I、D、H** | **I**/O / **D**ense storage / **H**igh throughput | ストレージ最適化 | 高性能なローカルストレージを提供。高いディスクスループットや大容量ローカルストレージが必要な環境向け。 |

# AMI（Amazon Machine Image）

EC2インスタンスを起動する際の **「型（テンプレート）」** のこと。  
「どのOS・ソフトウェア・ディスク構成でインスタンスを作るか」を定義したパッケージ。

## AMIに含まれるもの

| 要素 | 内容 |
| :--- | :--- |
| **OSイメージ** | Amazon Linux 2、Ubuntu、Windows Server など |
| **ルートボリューム** | OSがインストールされたディスクのスナップショット |
| **起動許可** | どのAWSアカウントがこのAMIを使ってよいか |
| **ブロックデバイスマッピング** | 起動時にアタッチするEBSボリュームの構成 |

## AMIの種類

| 種類 | 説明 |
| :--- | :--- |
| **AWS提供のAMI** | Amazon Linux 2023、Ubuntu、Windowsなど、AWSが管理する公式イメージ |
| **AWS Marketplace AMI** | サードパーティが構成済みアプリをパッケージ化して販売・配布するイメージ |
| **カスタムAMI** | 自分で立ち上げたインスタンスに設定を施し、それをイメージとして保存したもの |

## カスタムAMIの活用イメージ

```
① 通常のEC2を起動
↓
② ミドルウェアやアプリをインストール・設定
↓
③ そのインスタンスから「AMIを作成」（= スナップショット化）
↓
④ 次回からはこのAMIを使って即座に同構成のインスタンスを複製できる
```

**主なユースケース:**
- **Auto Scaling との組み合わせ** — 起動テンプレートにカスタムAMIを指定し、スケールアウト時に設定済み状態でインスタンスを自動追加
- **AMIリージョン間コピー** — AMIを別リージョンへコピーしてDR（災害復旧）構成に活用
- **ゴールデンイメージ** — セキュリティパッチや共通設定を焼き込んだ「正規の1枚」として管理

> **【試験ポイント】**  
> AMIは **リージョン単位** で管理される。別リージョンで使うには明示的にコピーが必要。  
> カスタムAMIから起動すれば、ブートストラップ（起動時スクリプト）の処理量を減らし **起動時間を短縮** できる。

# プレイスメントグループ（Placement Group）

EC2インスタンスを **「どのように物理ハードウェアに配置するか」** を制御する機能。  
目的に応じて3種類から選択する。

## 3種類の比較

| 種類 | 配置戦略 | ネットワーク | 耐障害性 | 主なユースケース |
| :--- | :--- | :---: | :---: | :--- |
| **クラスター**<br>（Cluster） | 同一AZ内の**同じラック**に集約 | 最高速・超低レイテンシ | 低い<br>（ラック障害で全滅リスク） | HPC、機械学習、ビッグデータのリアルタイム処理など高スループットが必要な場合 |
| **スプレッド**<br>（Spread） | インスタンスを**別々のラック**に分散<br>（1ラック1インスタンス） | 通常 | 最高<br>（ラック障害の影響を最小化） | 少数の重要インスタンスを確実に分離したい場合<br>※AZあたり **最大7インスタンス** の制限あり |
| **パーティション**<br>（Partition） | 複数の「パーティション（=ラック群）」に分割。<br>同パーティション内は同一ラック群、異パーティションは別ラック | 中程度 | 中程度<br>（パーティション単位で分離） | Hadoop、HBase、Kafkaなど分散システム。<br>パーティション情報をアプリが活用できる場合に最適 |

## イメージ図

```
【クラスター】          【スプレッド】            【パーティション】
  ┌──────────┐         ラック1  ラック2  ラック3    パーティション1  パーティション2
  │ ラック1  │          [EC2-A]  [EC2-B]  [EC2-C]   [EC2-A][EC2-B]  [EC2-C][EC2-D]
  │ [A][B][C]│          [EC2-D]  [EC2-E]  [EC2-F]    ↑同ラック群      ↑別ラック群
  └──────────┘         → 1ラックに1インスタンスのみ
  超高速・高密度        → 最も分散・最も安全
```

> **【試験ポイント】**  
> - **クラスター** → 「低レイテンシ」「高スループット」キーワードが出たらこれ。リスクとのトレードオフ。  
> - **スプレッド** → 「重要なインスタンスをハードウェア障害から分離」キーワードが出たらこれ。ただし **AZあたり7台上限** を覚えておく。  
> - **パーティション** → 「Hadoop/Kafka/HBase などの分散システム」キーワードが出たらこれ。

# Savings Plans 詳細

RIの弱点（柔軟性の低さ）を克服した後発の割引プラン。  
**「1時間あたりいくら使うか」を約束することで割引を受ける**モデル。

## 4種類の比較

| 種類 | 割引率 | 対象サービス | 縛り | 柔軟性 |
| :--- | :---: | :--- | :--- | :--- |
| **Compute SP** | 最大66% | EC2・Fargate・Lambda | 1年 or 3年 | **最高**。リージョン・OSなど全て自由 |
| **EC2 Instance SP** | 最大72% | EC2のみ | 1年 or 3年 | **中**。インスタンスファミリーとリージョンは固定（サイズ・OSは自由） |
| **SageMaker SP** | 最大64% | SageMaker のみ | 1年 or 3年 | **中**。ML開発・推論インスタンス向け |
| **Amazon Redshift SP** | 最大61% | Redshift のみ | 1年 or 3年 | **中**。クラスターの種類やリージョンをまたいで適用 |

## Savings Plans 購入時の2つの決断

```
① コミット金額（$/時間）を決める
   → 例：「1時間あたり $0.10 分の使用を約束します」

② 契約期間を決める
   → 1年契約 or 3年契約（3年の方が割引率が高い）
```

コミット金額を **超えた利用分** はオンデマンド料金で課金される。

## RIとの使い分け

| 比較軸 | Savings Plans | スタンダードRI |
| :--- | :--- | :--- |
| **柔軟性** | 高い（構成変更が容易） | 低い（OS・タイプ変更不可） |
| **キャパシティ予約** | ❌ なし | ✅ AZ指定時に自動付与 |
| **最大割引率** | 最大72%（EC2 Instance SP） | 最大72%（スタンダードRI） |
| **推奨ケース** | 構成が変わる可能性がある場合 | キャパシティ確保が最優先の場合 |

> **【試験ポイント】**  
> - **Savings Plans はキャパシティ予約ができない** → 確実にインスタンスを確保したい場合は「オンデマンドキャパシティ予約」または「スタンダードRI（AZ指定）」を使う。  
> - **Compute SP は3サービス（EC2・Fargate・Lambda）に横断的に適用** → コンテナやサーバーレスへの移行予定があっても無駄にならない。  
> - **EC2とFargateとLambdaを使ってコスト削減したい** → Compute SPの出番。
