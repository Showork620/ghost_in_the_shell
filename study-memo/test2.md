ElastiCache: RAMを使用するデータストア。RDSなどのDB連携において、負荷軽減と応答性能向上。

- ElastiCache for Memcached
  - キャッシュ。キャッシュ専用
  - フェイルオーバーはできない
  - 消える（DRできない）

- ElastiCache for Redis
  - キャッシュ。AZまたぐ。負荷分散になる。
  - フェイルオーバーできる
  - レプリカを作成する（DR適正）
    - 高可用性あり
    - バックアップあり
  - ユースケース
    - ログインのセッション管理
      - ログイン状態、カート情報
    - 商品ページの高速化
      - 商品詳細JSON、人気商品リスト
    - ゲームのランキング
      - user_id, score


### Cache Aside（最重要）
実質デフォルト戦略
キャッシュミス時だけDBアクセス
- コスト効率いい
- キャッシュ汚染が少ない

### Write Through
書き込み＝DB + キャッシュ同時更新
- 読み取りは常に高速
- 書き込み頻度が高いとボトルネック

###  Write Back（Write Behind）
まずキャッシュに書く → 後でDB
- 書き込み性能は最強
- 障害時にデータ消える可能性

TTLの追加
- TimeToLiveはキャッシュの一次データの有効期限

DynamoDB
- DynamoDBはAWS Lambdaと統合されている

DynamoDB Streams
- テーブルに対して行われた直近の24時間のの変更をログに保存する機能
- ストリームを参照することでいつどのようにテーブルを更新したかわかる
  - 非同期で動作、リアルタイムにアクセスできる
  - Lambdaと連携できるので、Streamの動きでイベント駆動できる

CloudTrail
- AWSリソースについて、いつ誰が操作して成功したかどうかのログ監査
- 誰が変更したか
- 変更を監視できるが、既存の状態を検出する機能はない

Config
- 変更後、リソースの状態がどう変わったか
- ルール違反していないか
- 既存の状態を検出する機能がある

CloudWatch Logs
- アプリ実行ログの本命



DAX
- DynamoDB用のCache Asideやってくれるやつ
  - アプリはDynamoを読む代わりにDAXを読ませる
  - Cache Aside（キャッシュがあればキャッシュを使う
  - 特徴
    - マイクロ秒単位のレイテンシ
    - 強い整合性はない
  - ユースケース
    - 広告入札
    - 金融トレード
    - ゲームランキング
    - リアルタイムスコア
  - 向かないもの
    - 強い整合性が必要なもの
      - 在庫管理、決済残高、注文確定直後の確定
    - 書き込みが多いもの
      - チャットメッセージ、ログ収集、IoTイベント
    - キャッシュヒット率が低い
      - 毎回違うデータを読むケース
