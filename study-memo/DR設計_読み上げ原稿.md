# DR設計（災害復旧）音声読み上げ用原稿

AWSのDR戦略は4段階あります。RTOが長い順（コストが低い順）に、バックアップ&リストア、パイロットライト、ウォームスタンバイ、マルチサイトです。この順番とRPO・RTOの関係はSAA試験で非常によく問われます。

---

## 1. マルチサイト（アクティブ/アクティブ）

RPOはほぼゼロ、RTOもほぼゼロ。4つの戦略の中でコストは最も高いです。金融や医療など、数秒の停止も許されないミッションクリティカルなシステムに使います。

### 関連サービス

**Route 53**。DNSレベルでトラフィックを複数リージョンに分散します。フェイルオーバールーティングポリシーとヘルスチェックを組み合わせて、異常リージョンを自動的に除外できます。試験ではルーティングポリシーの使い分け、フェイルオーバー、レイテンシーベース、加重ルーティングの違いが頻出です。

**Aurora Global Database**。最大5つのセカンダリリージョンにリアルタイムでレプリケーションし、通常1秒未満の遅延で同期します。障害時にはセカンダリリージョンを1分未満で昇格可能です。試験ではRDS Multi-AZとの違いが重要です。Multi-AZは同一リージョン内の高可用性、つまりHAであり、クロスリージョンDRにはAurora Global DBを選びます。

**RDS Multi-AZ**。同一リージョン内のAZ障害に対する高可用性を提供します。自動フェイルオーバーでスタンバイに切り替わりますが、クロスリージョンの災害復旧には対応しません。試験では「DR」ではなく「HA」の文脈であることに注意です。

**Global Accelerator**。AWSグローバルネットワークを使い、Anycast IPアドレスでトラフィックをルーティングします。Route 53との違いは、DNSキャッシュに依存しないため、フェイルオーバーがより高速な点です。試験ではRoute 53との比較が問われます。

**S3 クロスリージョンレプリケーション**。略してCRRです。バケット間でオブジェクトを自動的に別リージョンにレプリケートします。試験ではCRRにはバージョニングの有効化が必須であること、またS3 Replication Time Controlを使えば15分以内のレプリケーションがSLA保証されることが出題されます。

**DynamoDB Global Tables**。フルマネージドのマルチリージョン、マルチアクティブなデータベースです。各リージョンで読み書きが可能で、リアルタイムで双方向同期を行います。フェイルオーバー時に昇格操作が不要な点がAurora Global DBとの違いです。NoSQLのマルチサイトDRではこれが第一選択です。

**CloudFront**。グローバルCDNとしてエッジロケーションにコンテンツをキャッシュします。オリジンフェイルオーバー機能により、プライマリオリジン障害時にセカンダリオリジンへ自動で切り替わります。試験ではオリジングループの設定が問われます。

**ElastiCache Global Datastore**。ElastiCache for Redisのクロスリージョンレプリケーション機能です。セッション管理やキャッシュ層のDRとして出題される可能性があります。

## 2. ウォームスタンバイ

RPOは数秒、RTOは数分です。DRリージョンに縮小版の環境を常時稼働させておきます。コストは中程度から高程度です。数時間の停止が許容できないWEBサービスや業務システムに適用します。

パイロットライトとの最大の違いは、アプリケーションサーバーが常時稼働中である点です。パイロットライトではDBなどコアのみが稼働し、コンピュートは停止状態ですが、ウォームスタンバイでは全コンポーネントが縮小構成で動いています。そのため、フェイルオーバー時はスケールアップだけで済み、サーバーの起動を待つ必要がありません。

### 関連サービス

**EC2 Auto Scaling**。DRリージョンで縮小版のEC2フリートを常時稼働させ、フェイルオーバー時にスケールアップして本番負荷に対応します。試験では「最小台数で待機し、障害時にスケールアウト」というシナリオが頻出です。

**RDS リードレプリカ（クロスリージョン）**。DRリージョンにリードレプリカを配置し、データを継続的にレプリケーションします。障害時にリードレプリカをスタンドアロンDBへ昇格、英語でpromoteさせて書き込み可能にします。このpromote操作の理解は試験で問われます。

**Route 53 ヘルスチェック**。プライマリリージョンの異常を検知し、DNSフェイルオーバーでトラフィックをDRリージョンへ自動切り替えします。フェイルオーバールーティングポリシーとの組み合わせが試験で頻出です。

**Elastic Load Balancer**。DRリージョンのEC2フリートの前段でトラフィックを分散します。Route 53のエイリアスレコードと組み合わせ、リージョン単位のフェイルオーバーを実現します。

**CloudWatch アラーム**。プライマリリージョンの異常を検知し、Auto ScalingポリシーやSNS通知をトリガーします。Route 53ヘルスチェックと連携して自動フェイルオーバーの判断材料となります。

**Aurora Global Database**。ストレージレベルでクロスリージョンレプリケーションを行い、RPO 1秒未満、RTO 1分未満を実現します。RDSクロスリージョンリードレプリカよりも高速なフェイルオーバーが可能です。試験ではRDSリードレプリカとの使い分けが問われます。

**AWS Global Accelerator**。Anycast IPアドレスによりDNS TTLに依存しない高速フェイルオーバーを実現します。トラフィックダイヤルで各リージョンへのトラフィック割合を制御できます。

**AWS Elastic Disaster Recovery（AWS DRS）**。ブロックレベルの継続的レプリケーションにより、ソースサーバーをAWSへDRします。オンプレミスからAWSへのDRシナリオで出題されます。旧CloudEndure Disaster Recoveryの後継サービスです。

**CloudFormation StackSets**。DRリージョンのインフラをIaCで定義し、マルチリージョンに一括展開します。環境の一貫性を維持し、スケールアップ時の追加リソースを迅速にプロビジョニングします。

## 3. パイロットライト

RPOは数分、RTOは数十分です。コストは中程度です。データベースなどコアコンポーネントのみをDRリージョンで常時稼働させ、アプリケーションサーバーなどのコンピュートリソースは停止状態で待機します。中規模システムに適用します。

ウォームスタンバイとの違いは、アプリサーバーが停止していることです。フェイルオーバー時にはサーバーの起動とスケールアウトが必要なため、ウォームスタンバイよりRTOが長くなります。試験で「最小コストで」「コア部分のみ稼働」というキーワードが出たらパイロットライト、「すぐにトラフィックを受けられる」「常時稼働」とあればウォームスタンバイです。

### 関連サービス

**RDS クロスリージョンレプリカ**。パイロットライトの「常に火を灯しておく」中核です。DRリージョンにリードレプリカを非同期レプリケーションで維持し、障害時にプライマリへ昇格させます。Aurora Global Databaseを使えばRPOは1秒未満、昇格RTOは1分以内に短縮可能です。

**AMI（Amazon Machine Image）**。EC2 Image Builderなどで最新のゴールデンAMIを作成し、DRリージョンへコピーしておきます。障害時にこのAMIからEC2を即座に起動します。古いAMIでは復旧後にパッチ適用が必要になり、RTOが悪化します。

**EC2（停止状態で待機）**。パイロットライトではアプリケーションサーバーは停止状態です。障害時に起動し、Auto Scalingでスケールアウトします。Auto Scaling Groupの最小容量を0に設定し、フェイルオーバー時に引き上げるのが典型パターンです。

**CloudFormation**。DRリージョンのインフラ全体をIaCで定義します。障害時のスケールアップや追加リソースのプロビジョニングを自動化します。パイロットライトの成功は自動化の徹底にかかっています。

**Route 53**。フェイルオーバールーティングとヘルスチェックの組み合わせでトラフィックをDRリージョンへ切り替えます。Route 53はデータプレーン操作であり、コントロールプレーン障害の影響を受けにくいという点が試験で問われます。

**S3 クロスリージョンレプリケーション**。静的アセットやログ、バックアップデータをDRリージョンへ非同期で自動複製します。S3 Replication Time Controlを有効にすれば15分以内にレプリケーションが完了します。

**AWS DRS（Elastic Disaster Recovery）**。オンプレミスや他クラウドのサーバーを、ブロックレベルで継続的にAWSへレプリケーションします。ステージングエリアで低コストのインスタンスを使い、フェイルオーバー時にフルスペックで起動します。

**AWS Application Recovery Controller（ARC）**。フェイルオーバーの「スイッチ」としてレディネスチェックとルーティングコントロールを提供します。手動でのフェイルオーバー開始を安全に制御し、誤ったフェイルオーバーを防止します。

## 4. バックアップ&リストア

RPOは数時間、RTOは数時間から1日です。4つの戦略の中でコストは最も低いです。開発・検証環境や重要度の低い社内システムに適用します。稼働中のDRリソースが不要で、ストレージコストのみで済みます。

### 関連サービス

**Amazon S3（バックアップ保存先）**。バックアップデータの保存先として最も基本的なサービスです。イレブンナインの耐久性を持ちます。S3 Glacier、Glacier Deep Archiveを使えば長期保存コストを大幅に削減できます。ただし取り出しに時間がかかるためRTOに影響します。試験ではストレージクラスの使い分けとライフサイクルポリシーによる自動階層化が出題されます。

**RDS 自動スナップショット**。自動バックアップは最大35日間保持可能です。ポイントインタイムリカバリ、略してPITRにより、5分間隔の粒度でデータベースを任意の時点に復元できます。クロスリージョンへのスナップショットコピーも可能です。試験ではPITRのRPOが約5分であること、手動スナップショットには保持期間の制限がないことが問われます。

**EBS スナップショット**。EBSボリュームの増分スナップショットをS3に保存します。初回はフルバックアップ、以降は変更分のみです。Amazon Data Lifecycle Manager、略してDLMで作成・保持・削除を自動化できます。試験では増分バックアップであること、DLMによる自動化、クロスリージョンコピーが可能であることが問われます。

**AWS Backup**。EBS、EFS、RDS、DynamoDB、S3、FSxなど複数サービスのバックアップを一元的に管理・自動化するフルマネージドサービスです。バックアッププランでスケジュールと保持ポリシーを定義します。Vault Lockで削除不可にすることでコンプライアンス要件にも対応します。クロスリージョン・クロスアカウントバックアップにも対応しています。試験では複数サービスの一元管理が最大の利点として出題されます。

**CloudFormation（環境再構築）**。IaCによりインフラ環境をテンプレートとして定義します。DR時にはテンプレートから別リージョンに環境を迅速に再構築できます。バックアップ&リストア戦略においてRTOを短縮する手段として出題されます。

**DynamoDB（PITRとオンデマンドバックアップ）**。ポイントインタイムリカバリを有効にすると、過去35日間の任意の秒単位で復元可能です。注意として、PITRはデフォルトで無効なので、手動で有効化が必要です。

**Route 53（DNSフェイルオーバー）**。バックアップ&リストアでも、復旧後のトラフィック切り替えにRoute 53のフェイルオーバールーティングを使います。DR構成の問題ではほぼ必ずRoute 53が登場します。

**Amazon EFS**。AWS Backupと連携してファイルシステムのバックアップが可能です。クロスリージョンレプリケーション機能もあります。試験ではEBSとの違い、EBSは単一AZ、EFSはマルチAZで複数EC2から同時マウント可能という点が問われます。

**Amazon FSx**。FSx for Windows File ServerやFSx for NetApp ONTAPのバックアップをAWS Backupで管理できます。オンプレミスのWindows環境からの移行・DRシナリオで出題されます。SMBプロトコル対応でActive Directory統合が可能です。

---

## まとめ：4戦略の比較

| 戦略 | RPO | RTO | コスト | DRリージョンの状態 |
|------|-----|-----|--------|-----------------|
| バックアップ&リストア | 数時間 | 数時間〜1日 | 最低 | 稼働リソースなし、バックアップのみ |
| パイロットライト | 数分 | 数十分 | 中 | DBなどコアのみ稼働、コンピュートは停止 |
| ウォームスタンバイ | 数秒 | 数分 | 中〜高 | 全コンポーネントが縮小構成で稼働 |
| マルチサイト | ほぼゼロ | ほぼゼロ | 最高 | 本番同等の環境がアクティブに稼働 |

試験では「コストとRTOのトレードオフ」「要件に応じた最適な戦略の選択」が必ず問われます。また、データプレーンとコントロールプレーンの区別も重要です。フェイルオーバー操作にはRoute 53などのデータプレーンを使い、CloudFormationなどのコントロールプレーンは障害時に利用できない可能性があることを覚えておきましょう。
