# Well-Architected Framework 6つの柱 — 関連AWSサービス（SAA試験対応）

音声読み上げ用原稿。各柱ごとに、AWS SAA試験で出題されるサービスとその役割をまとめる。

---

## 1. セキュリティ

### IDとアクセス管理

- **IAM**: AWSリソースへのアクセスをユーザー、グループ、ロール、ポリシーで制御する基盤サービス。最小権限の原則を実現する。試験ではポリシーの評価ロジック（明示的Deny優先）やクロスアカウントのAssumeRoleが頻出。
- **IAM Identity Center（旧AWS SSO）**: 複数AWSアカウントへのシングルサインオンを一元管理する。AWS Organizationsと連携したマルチアカウント環境のアクセス管理に最適。
- **Amazon Cognito**: Webアプリやモバイルアプリのエンドユーザー認証を提供する。ユーザープールでサインイン、IDプールでAWSリソースへの一時認証情報を発行する。外部IDプロバイダとの連携が試験に出る。
- **AWS STS**: IAMロールの引き受けやフェデレーションで一時的なセキュリティ認証情報を発行する。クロスアカウントアクセスやEC2のインスタンスプロファイルでの利用が問われる。
- **AWS Directory Service**: オンプレミスのActive DirectoryとAWS環境を統合する。AWS Managed Microsoft ADとAD Connectorの使い分けが試験に出る。

### 検出と監視

- **AWS CloudTrail**: AWSアカウント内の全APIコールを記録し、誰がいつ何をしたかの監査証跡を提供する。全リージョンでの有効化とS3へのログ集約、CloudTrail Insightsによる異常検知が試験に出る。
- **Amazon GuardDuty**: VPCフローログ、CloudTrailログ、DNSログを機械学習で分析し、悪意あるアクティビティを自動検出する脅威検出サービス。有効化するだけで動作する点が重要。
- **Amazon Inspector**: EC2インスタンスやECRコンテナイメージの脆弱性を自動スキャンする。脆弱性評価の自動化ソリューションとして出題される。
- **AWS Config**: AWSリソースの構成変更を継続的に記録し、Configルールでコンプライアンス準拠状況を自動チェックする。非準拠リソースの自動修復との連携が頻出。
- **AWS Security Hub**: GuardDuty、Inspector、Config、Macieなど複数のセキュリティサービスの検出結果を一元集約するダッシュボード。マルチアカウント環境でのセキュリティ統合管理に最適。
- **Amazon Detective**: GuardDutyの検出結果を起点にセキュリティイベントの根本原因を可視化・調査するサービス。

### データ保護

- **AWS KMS**: 暗号化キーの作成、管理、ローテーションを一元管理する。S3、EBS、RDSなど多くのサービスと統合し、保存データの暗号化を実現する。CMKとAWSマネージドキーの使い分け、キーポリシーによるアクセス制御が試験に出る。
- **AWS CloudHSM**: 専用のハードウェアセキュリティモジュールで暗号化キーを管理する。FIPS 140-2レベル3準拠が求められる場合に使う。KMSとの使い分けが出題される。
- **AWS Certificate Manager（ACM）**: SSL/TLS証明書のプロビジョニング、管理、自動更新を行い、通信中のデータを暗号化する。ALBやCloudFrontでのHTTPS通信の実現方法として出題される。
- **Amazon Macie**: S3バケット内のデータを機械学習でスキャンし、個人情報やクレジットカード番号などの機密データを検出する。S3の機密データ自動検出として出題される。
- **AWS Secrets Manager**: データベースの認証情報やAPIキーなどのシークレットを安全に保存し、自動ローテーションを提供する。RDSのパスワード自動ローテーションが試験に出る。
- **Systems Manager Parameter Store**: 設定値やシークレットをセキュアに一元管理する。Secrets Managerとの違いは自動ローテーション機能の有無。低コストでシークレット管理したい場合に選択する。

### ネットワーク・インフラ保護

- **Amazon VPC**: 論理的に分離されたネットワーク空間を提供する。パブリック/プライベートサブネットの設計と、VPCエンドポイント（ゲートウェイ型とインターフェース型）が頻出。
- **セキュリティグループ（SG）**: インスタンスレベルのステートフルファイアウォール。デフォルトで全インバウンド拒否、全アウトバウンド許可。戻りトラフィックは自動許可される。
- **ネットワークACL（NACL）**: サブネットレベルのステートレスファイアウォール。SGとの違いはステートレスであること、ルール番号順評価、明示的拒否が可能なこと。特定IPのブロックにはNACLを使う。
- **AWS WAF**: SQLインジェクションやXSSなどのWebアプリケーション攻撃から保護する。ALB、CloudFront、API Gatewayと連携する。レート制限やIPベースのアクセス制御が出題される。
- **AWS Shield**: DDoS攻撃からの保護。Standardは無料で自動適用、Advancedは高度なDDoS保護と24時間365日のDDoS対応チーム（DRT）へのアクセスを含む。StandardとAdvancedの違いが頻出。
- **AWS Firewall Manager**: Organizations全体でWAF、Shield Advanced、SGのルールを一元管理する。複数アカウントにわたるセキュリティポリシーの一括管理が必要な場合に選択する。
- **AWS Network Firewall**: VPCレベルでステートフルなパケットインスペクション、侵入検知・防止（IDS/IPS）を提供するマネージドファイアウォール。
- **AWS PrivateLink**: VPCエンドポイントを通じてAWSサービスやサードパーティサービスへのトラフィックをAWSネットワーク内に保つ。インターネットを経由しないプライベート接続を実現する。

### イベント対応と自動化

- **Amazon EventBridge**: AWSサービスのイベントをリアルタイムに検知し、LambdaやSNSに自動ルーティングする。GuardDutyの検出結果やConfig非準拠をトリガーにした自動修復パイプラインが出題される。
- **AWS Lambda**: イベント駆動でセキュリティ対応を自動化する。非準拠リソースの自動修復や不審なアクティビティへの自動対応を実現する。

### ガバナンスとコンプライアンス

- **AWS Organizations**: マルチアカウント環境を一元管理する。SCP（サービスコントロールポリシー）で組織全体にセキュリティガードレールを強制する。SCPによるサービスやリージョンの制限が頻出。
- **AWS Control Tower**: マルチアカウント環境のセットアップをベストプラクティスに基づいて自動化し、ガードレール（予防的・検出的）を適用する。
- **AWS Artifact**: AWSのコンプライアンスレポート（SOC、PCI DSSなど）にオンデマンドでアクセスできる。監査対応で必要な場合に使用する。
- **AWS Trusted Advisor**: セキュリティグループの過度に開放されたポート、MFAの未設定、S3のパブリックアクセスなど、ベストプラクティスに反する設定を自動チェックする。

## 2. 信頼性

### コンピューティング

- **Amazon EC2**: マルチAZ配置で冗長化し、障害時に別AZのインスタンスで処理を継続する。マルチAZ構成と起動テンプレートの組み合わせが問われる。
- **EC2 Auto Scaling**: 需要増減に応じてインスタンス数を自動調整し、過負荷による障害を防止する。スケーリングポリシーの種類（ターゲット追跡、ステップ、簡易）の選択が出題される。
- **AWS Lambda**: サーバーレスで自動スケールし、インフラ障害の管理が不要。障害時の自動復旧アクションやイベント駆動型アーキテクチャで出題される。
- **AWS Fargate**: コンテナのサーバーレス実行環境で、マルチAZ配置によりインフラ管理なしに信頼性を確保する。

### ネットワーキング

- **Elastic Load Balancing（ALB/NLB）**: トラフィックを複数AZの正常なターゲットに分散し、障害インスタンスを自動的に切り離す。ヘルスチェック設定とクロスゾーン負荷分散が頻出。
- **Amazon Route 53**: DNSレベルのヘルスチェックとフェイルオーバールーティングで、リージョン障害時にトラフィックを切り替える。フェイルオーバー、加重、レイテンシーベースのルーティングポリシーが出題される。
- **AWS Global Accelerator**: AWSグローバルネットワーク経由でトラフィックを最適なエンドポイントにルーティングし、リージョン障害時に自動フェイルオーバーする。Route 53との使い分けが出題される。
- **Amazon CloudFront**: エッジロケーションからコンテンツを配信し、オリジン障害時もキャッシュから応答を返せる。オリジンフェイルオーバー構成が出題される。

### データベース

- **Amazon RDS（Multi-AZ）**: 同期レプリケーションによるスタンバイ配置で、プライマリ障害時に自動フェイルオーバーする。Multi-AZとリードレプリカの違い、自動バックアップとPITR（ポイントインタイムリカバリ）が頻出。
- **Amazon Aurora**: 3つのAZに6つのデータコピーを自動保持し、最大15のリードレプリカで高可用性を実現する。Aurora Global Databaseでマルチリージョン構成も可能。
- **Amazon DynamoDB**: 3つのAZにデータを自動複製する。グローバルテーブルでマルチリージョンの読み書きを実現する。グローバルテーブルとオンデマンドキャパシティ、PITRが出題される。
- **Amazon ElastiCache**: Redisのレプリカノードによるフェイルオーバーでキャッシュ層の高可用性を確保する。RedisのMulti-AZレプリケーションとMemcachedの使い分けが出題される。

### ストレージ

- **Amazon S3**: 最低3つのAZにオブジェクトを自動複製し、イレブンナインの耐久性を提供する。バージョニング、クロスリージョンレプリケーション、ストレージクラスが頻出。
- **Amazon EBS**: AZ内で自動複製され、スナップショットによるバックアップとリージョン間コピーが可能。スナップショットの増分バックアップが出題される。
- **Amazon EFS**: マルチAZに自動保存される共有ファイルストレージ。複数EC2インスタンスからの同時アクセスが可能。

### バックアップ・災害復旧

- **AWS Backup**: 複数AWSサービスのバックアップを一元管理・自動化する。クロスリージョンコピーやクロスアカウントバックアップが出題される。
- **AWS Elastic Disaster Recovery（DRS）**: サブ秒のRPOで継続的データ保護を行い、リージョン間フェイルオーバーを実現する。DR戦略の4パターン（バックアップ、パイロットライト、ウォームスタンバイ、マルチサイト）との関連が出題される。
- **AWS Storage Gateway**: オンプレミスのデータをAWSにバックアップするゲートウェイ。File、Volume、Tapeゲートウェイの使い分けが出題される。
- **AWS DataSync**: オンプレミスからAWSへのデータ転送を自動化する。Storage Gatewayとの使い分けが出題される。

### メッセージング・疎結合

- **Amazon SQS**: コンポーネント間を疎結合にし、一方の障害が全体に波及することを防ぐ。標準キューとFIFOキューの違い、デッドレターキュー、可視性タイムアウトが頻出。
- **Amazon SNS**: Pub/Subメッセージングで障害通知やイベント駆動アーキテクチャの信頼性を確保する。SQSとの連携（ファンアウトパターン）が出題される。
- **Amazon EventBridge**: イベント駆動型アーキテクチャのハブとして、サービス間の疎結合と自動化された障害対応を実現する。
- **Amazon Kinesis Data Streams**: リアルタイムデータストリーミングで、データの順序保証と複数コンシューマーへの配信を信頼性高く行う。
- **Amazon MQ**: ActiveMQ/RabbitMQのマネージド版。レガシーアプリのクラウド移行時にSQS/SNSとの使い分けが出題される。

### モニタリング

- **Amazon CloudWatch**: メトリクス監視、アラーム設定、ログ集約で障害の早期検知と自動復旧をトリガーする。アラームアクション（EC2復旧、Auto Scaling）が頻出。
- **AWS X-Ray**: 分散アプリケーションのトレースで障害の根本原因を特定する。マイクロサービス環境でのレイテンシ分析が出題される。
- **AWS Trusted Advisor / Service Quotas**: サービス制限の監視と管理で、リソース上限到達による障害を事前に防止する。

## 3. パフォーマンス効率

### コンピューティング

- **Amazon EC2**: 多様なインスタンスファミリー（汎用、コンピューティング最適化、メモリ最適化、ストレージ最適化、高速コンピューティング）からワークロードに最適なタイプを選択する。メモリ集約型ならRシリーズ、CPU集約型ならCシリーズなど。
- **AWS Lambda**: サーバーレスでイベント駆動型の短時間実行ワークロードに最適。インフラ管理のオーバーヘッドをゼロにする。
- **AWS Fargate**: コンテナをサーバーレスで実行し、EC2インスタンスの管理なしにコンテナワークロードを最適なリソース量で動かす。
- **EC2 Auto Scaling**: 需要に応じてインスタンス数を自動調整する。ターゲット追跡、予測スケーリング、スケジュールスケーリングの使い分けが出題される。
- **AWS Graviton プロセッサ**: AWS独自開発のARMベースプロセッサ。従来のx86インスタンスと比較して最大40%のコスト対性能比の向上を実現する。

### ストレージ

- **Amazon EBS**: ボリュームタイプ（gp3、io2、st1、sc1）を適切に選択してIOPSとスループット要件を満たす。gp3とio2の違い、プロビジョンドIOPSが必要なユースケースが出題される。
- **Amazon EFS**: 複数EC2インスタンスから同時アクセス可能な共有ファイルシステム。パフォーマンスモード（汎用/最大I/O）の選択が出題される。
- **Amazon FSx**: FSx for Lustreは HPC やML向けの高性能並列ファイルシステム。S3と連携した高速ファイルアクセスが必要な場合に選択する。
- **Amazon S3 Transfer Acceleration**: CloudFrontのエッジロケーションを利用して遠距離からのS3へのデータ転送速度を向上させる。マルチパートアップロードとの組み合わせが出題される。

### データベース

- **Amazon Aurora**: MySQL/PostgreSQL互換で、標準RDSの最大5倍の性能。最大15個のリードレプリカとAurora Serverlessによるオンデマンドスケーリングを提供する。
- **Amazon DynamoDB**: どんなスケールでも一桁ミリ秒のレイテンシを実現するNoSQLデータベース。DynamoDB Accelerator（DAX）でマイクロ秒レベルのキャッシュ読み取りも可能。
- **Amazon ElastiCache**: Redis/Memcachedベースのインメモリキャッシュでデータベースの負荷を軽減し、レスポンスをマイクロ秒単位に高速化する。RedisとMemcachedの選択基準が出題される。
- **Amazon Redshift**: ペタバイト規模のデータウェアハウス。列指向ストレージと大規模並列処理で分析クエリを高速実行する。Redshift Spectrumを使ったS3直接クエリも出題される。
- **Amazon OpenSearch Service（旧Elasticsearch）**: 全文検索やログ分析をリアルタイムで高速処理するマネージド検索・分析サービス。

### ネットワーキングとコンテンツ配信

- **Amazon CloudFront**: グローバルCDNでエッジロケーションからコンテンツを配信し、ユーザーへのレイテンシを大幅に低減する。S3やALBをオリジンとした構成が頻出。
- **Amazon Route 53**: レイテンシベースルーティングや地理的近接性ルーティングで、ユーザーを最も応答の速いエンドポイントに誘導する。
- **AWS Global Accelerator**: AWSグローバルネットワークでネットワーク距離を短縮しジッターを減少させる。CloudFrontとの使い分け（CDNキャッシュ vs ネットワーク最適化）が出題される。
- **Elastic Load Balancing（ALB/NLB）**: トラフィックを複数ターゲットに分散しスループットを向上させる。ALBはHTTP/HTTPS、NLBはTCP/UDPの超低レイテンシ負荷分散。
- **AWS Direct Connect**: オンプレミスとAWS間の専用回線接続で、一貫した低レイテンシ・高帯域幅を実現する。VPN接続との比較が出題される。
- **VPCエンドポイント**: S3やDynamoDB等へのトラフィックをAWS内部ネットワークで完結させレイテンシを削減する。ゲートウェイエンドポイントとインターフェースエンドポイントの使い分けが出題される。
- **Amazon API Gateway**: キャッシュ機能やスロットリングでバックエンドの負荷を制御しつつ低レイテンシなAPI応答を実現する。Lambdaとの統合によるサーバーレスAPI構成が出題される。

### モニタリングと最適化

- **Amazon CloudWatch**: カスタムメトリクスの活用とアラームによるAuto Scaling連携が出題される。
- **AWS X-Ray**: 分散アプリケーションのリクエストトレーシングでパフォーマンスのボトルネックを特定する。
- **AWS Compute Optimizer**: EC2、Lambda、EBSのパフォーマンスデータを分析し最適なリソースサイズを推奨する。

### データ転送と統合

- **Amazon SQS**: メッセージキューで処理の並列化やバースト対応を実現しスループットを向上させる。標準キューとFIFOキューの性能特性の違いが出題される。
- **Amazon Kinesis**: リアルタイムストリーミングデータの取り込み・処理を低レイテンシで実現する。Data StreamsとData Firehoseの使い分けが重要。
- **Amazon SNS**: ファンアウトパターンで一つのメッセージを複数サブスクライバーに同時配信し処理を並列化する。SNSとSQSのファンアウト構成が出題される。
- **AWS Step Functions**: サーバーレスワークフローのオーケストレーションで並列実行やエラーハンドリングを最適化する。
- **AWS Snow Family（Snowball等）**: テラバイト〜ペタバイト級の大量データの物理転送でネットワーク帯域幅のボトルネックを回避する。

## 4. コスト最適化

### コスト管理・可視化

- **AWS Cost Explorer**: 過去と現在のコスト・使用量を可視化し、将来のコスト予測も可能。Savings PlanやRIの購入推奨も提示する。
- **AWS Budgets**: 月次予算を設定し、閾値超過時のアラート通知を行う。コスト超過を未然に防ぐ手段として出題される。
- **AWS Cost and Usage Report（CUR）**: 最も詳細なコスト・使用量データを提供する。最も粒度の細かいコストデータが必要な場面で正解になる。
- **AWS Cost Anomaly Detection**: 機械学習で異常なコスト急増を自動検出しアラートを送信する。
- **AWS Pricing Calculator**: AWSサービスの見積もり計算ツール。移行前のコスト見積もりに使用する。

### 購入オプション・料金モデル

- **Savings Plans**: 1年または3年の一定使用量コミットメントで最大72%割引。Compute Savings Plansはインスタンスファミリー・リージョン問わず柔軟に適用。Instance Savings Plansはさらに高い割引率だが柔軟性は低い。RIとの違いが頻出。
- **Reserved Instances（RI）**: EC2、RDS、ElastiCache、Redshiftなどで利用可能な1年/3年のコミットメント割引。安定した長期ワークロードに対するコスト削減策として定番。
- **EC2 Spot Instances**: 未使用のEC2容量を最大90%割引で利用可能。2分前の中断通知がある。バッチ処理やCI/CDなど中断耐性のあるワークロード向け。

### コンピューティング

- **EC2 Auto Scaling**: 需要に応じてインスタンス数を自動増減し、過剰プロビジョニングを防止する。需要変動がある場合のコスト最適化と可用性の両立で頻出。
- **AWS Lambda**: 実行時間とリクエスト数に対してのみ課金。アイドル時のコストがゼロ。低頻度・不定期な処理のコスト効率化で出題される。
- **AWS Fargate**: 使用したvCPUとメモリに対してのみ課金。EC2管理コストを削減するシナリオで出題される。
- **AWS Graviton**: x86と比較して最大40%のコストパフォーマンス向上を提供するプロセッサ。同等性能をより低コストで実現する選択肢として出題される。

### ストレージ

- **Amazon S3（ストレージクラス）**: Standard、Standard-IA、One Zone-IA、Glacier Instant Retrieval、Glacier Flexible Retrieval、Glacier Deep Archive、Intelligent-Tieringなど、アクセス頻度に応じた最適なストレージ層を選択する。ライフサイクルポリシーでの自動移行が頻出。
- **S3 Intelligent-Tiering**: アクセスパターンを自動分析し最もコスト効率の良い層に自動移動する。アクセス頻度が予測できない場合に最適。
- **Amazon EBS**: gp3はgp2より安価で性能調整が可能。不要なボリュームやスナップショットの削除がコスト削減につながる。
- **Amazon EFS（IA）**: Infrequent Accessストレージクラスでアクセス頻度の低いファイルのコストを最大92%削減できる。

### データベース

- **Amazon Aurora Serverless**: 使用量に応じて自動スケーリングし、アイドル時は一時停止も可能。開発・テスト環境や可変ワークロードに最適。
- **Amazon DynamoDB**: オンデマンドモードで使った分だけ課金、プロビジョンドモードでRIを適用可能。モードの使い分けが出題される。
- **Amazon ElastiCache**: キャッシュ導入でDB負荷を軽減し、RDSのインスタンスサイズを小さくできるため間接的にコスト削減になる。

### ネットワーク

- **Amazon CloudFront**: エッジキャッシュでオリジンへのリクエスト数とデータ転送量を削減し、S3やALBからのデータ転送コストを下げる。S3からの大量配信のコスト削減策として頻出。
- **VPCエンドポイント（Gateway型）**: S3やDynamoDBへのトラフィックをAWS内部に閉じ、NATゲートウェイのデータ処理料金を削減する。Gateway Endpointは無料。NATゲートウェイのコスト回避策として頻出。
- **Elastic IP / NAT Gateway**: 未使用のElastic IPには課金が発生する。NATゲートウェイはデータ処理量に比例して高額になる。不要なら削除すべき。

### ガバナンス

- **AWS Organizations**: 一括請求（Consolidated Billing）でボリュームディスカウントを得られる。RI/SPの共有も可能。
- **リソースタグ付け / コスト配分タグ**: タグでコストをプロジェクト・部門ごとに分類・追跡する。部門別のコスト追跡方法として出題される。
- **AWS Trusted Advisor**: 未使用リソースや低使用率リソースの特定。コスト削減の推奨事項を得る方法として頻出。
- **AWS Compute Optimizer**: 機械学習で過剰プロビジョニングを検出し適切なサイズを推奨する。ライトサイジングのシナリオで出題される。

## 5. 運用上の優秀性

### Infrastructure as Code（IaC）

- **AWS CloudFormation**: JSON/YAMLテンプレートでインフラをコードとして定義し、環境を再現可能にする。テンプレートによるスタック管理、ドリフト検出、変更セットが出題される。
- **AWS CDK**: プログラミング言語でCloudFormationテンプレートを生成するフレームワーク。CloudFormationの上位レイヤーとして出題される。
- **AWS SAM**: サーバーレスアプリケーションのデプロイをCloudFormationの拡張として簡潔に記述する。

### CI/CD

- **AWS CodePipeline**: ソースからビルド、テスト、デプロイまでを自動化するCI/CDパイプライン。小さく頻繁なリリースでリスクを低減する。
- **AWS CodeBuild**: ソースコードのコンパイル、テスト実行、パッケージ作成を行うフルマネージドなビルドサービス。
- **AWS CodeDeploy**: EC2、Lambda、ECSへのデプロイを自動化する。ローリング、ブルー/グリーン、カナリアのデプロイ戦略の違いと使い分けが出題される。

### モニタリング / オブザーバビリティ

- **Amazon CloudWatch**: メトリクス収集、ログ集約、アラーム設定、ダッシュボードを一元的に行う。カスタムメトリクス、CloudWatch Logs、メトリクスフィルターが頻出。
- **AWS X-Ray**: 分散アプリケーションのリクエストをトレースしボトルネックやエラーの根本原因を可視化する。
- **AWS CloudTrail**: 全API呼び出しを記録・監査する。管理イベントとデータイベントの違い、S3へのログ保存が頻出。
- **Amazon EventBridge**: AWSサービスやカスタムアプリのイベントを検出しルールに基づいてルーティングする。イベント駆動アーキテクチャの構築が出題される。
- **VPC Flow Logs**: VPC内のネットワークトラフィックを記録する。ログの出力先（CloudWatch Logs、S3）が出題される。

### 運用自動化 / 構成管理

- **AWS Systems Manager**: EC2やオンプレミスサーバーの構成管理、パッチ適用、コマンド実行、ランブック自動化を一元的に行う。Session Manager（SSHなしアクセス）、Parameter Store、Patch Manager、Run Commandの使い分けが頻出。
- **AWS Config**: リソースの構成変更を継続的に記録し、Configルールでコンプライアンスを自動チェックする。リメディエーション（自動修復）アクションが出題される。

### インシデント対応

- **AWS Lambda**: CloudWatchアラームやEventBridgeと連携して運用イベントへの自動応答処理を実装する。イベント駆動の自動修復パターンが頻出。
- **Amazon SNS**: アラームやイベントの通知を関係者やシステムに配信する。CloudWatchアラームの通知先としてのエスカレーションに活用する。
- **AWS Health Dashboard**: AWSリソースに影響を与えるイベントや障害をリアルタイムで通知する。Service Health DashboardとPersonal Health Dashboardの違いが出題される。

### ガバナンス

- **AWS Organizations**: 複数アカウントを一元管理し、SCPでガバナンスを適用する。OUの階層設計と一括請求が頻出。
- **AWS Control Tower**: マルチアカウント環境のセットアップとガバナンスを自動化する。ランディングゾーンとガードレールの概念が出題される。
- **AWS Trusted Advisor**: 5つの観点（コスト、パフォーマンス、セキュリティ、耐障害性、サービス制限）からAWS環境を自動チェックする。
- **AWS Service Catalog**: 承認済みITサービスのカタログを作成し、標準化されたリソースをセルフサービスで提供する。

### ログ分析

- **Amazon Athena**: S3上のデータにSQLクエリを直接実行するサーバーレス分析サービス。CloudTrailログやALBログの分析にコスト効率が良い方法として頻出。
- **AWS Glue**: S3のログデータを検出・変換し分析用に準備するETLサービス。Data Catalogの役割が出題される。
- **Amazon QuickSight**: BIダッシュボードを作成し運用データを可視化する。Athenaとの連携で出題される。

## 6. 持続可能性

### コンピューティング

- **AWS Graviton プロセッサ（ARM）**: x86と比較してワット当たりの性能が大幅に高く、同じ処理をより少ない電力で実行できる。EC2、RDS、ElastiCache等でGravitonインスタンスを選択する。
- **AWS Lambda**: サーバーレスでアイドル時のリソース消費がゼロ。ハードウェア保持の無駄を排除する。サーバーレスアーキテクチャが持続可能性のベストプラクティス。
- **AWS Fargate**: コンテナ向けサーバーレスコンピュートで、必要な分だけリソースを使用しインフラの無駄を削減する。
- **EC2 Auto Scaling**: 需要に応じてインスタンス数を動的に拡縮し、過剰プロビジョニングを防止してアイドルリソースのエネルギー消費を削減する。
- **EC2 Spot Instances**: AWS内の未使用キャパシティを再利用するため、既存リソースの利用効率を最大化する。
- **AWS Compute Optimizer**: 機械学習で最適なインスタンスタイプとサイズを推奨し、過剰プロビジョニングによるエネルギー浪費を防ぐ。

### サーバーレス/マネージドデータベース

- **Amazon Aurora Serverless**: 容量が自動スケーリングし、未使用時はリソースを解放するためアイドル時のエネルギー消費を最小化する。
- **Amazon DynamoDB**: オンデマンドモードで需要に応じたリソース配分を実現しエネルギー効率を最適化する。
- **Amazon ElastiCache**: インメモリキャッシュでバックエンドDBへのクエリ回数を削減し、コンピュートリソースの消費を抑える。
- **Amazon Athena**: サーバーレスクエリサービスで常時稼働クラスタが不要。アイドルリソースを排除する。

### ストレージ

- **Amazon S3（ライフサイクル / Intelligent-Tiering）**: アクセス頻度の低いデータを低エネルギーのストレージクラスに自動移行する。
- **Amazon S3 Glacier**: 長期アーカイブ用ストレージで、不要データをエネルギー効率の高い層に保管し環境負荷を低減する。
- **Amazon Data Lifecycle Manager**: EBSスナップショットやAMIの作成・保持・削除を自動化し、不要なデータコピーの蓄積を防止する。
- **Amazon EBS**: ボリュームサイズやタイプを動的に変更でき、過剰なストレージプロビジョニングを削減する。

### ネットワーク/コンテンツ配信

- **Amazon CloudFront**: エッジロケーションでコンテンツをキャッシュし、オリジンへのリクエスト数とデータ転送量を削減して全体のエネルギー消費を抑える。
- **Lambda@Edge / CloudFront Functions**: エッジロケーションで軽量な処理を実行し、オリジンへの不要なラウンドトリップを削減する。
- **Amazon Route 53**: レイテンシベースや地理的ルーティングでユーザーに最も近いリソースに誘導し、不要なネットワーク転送を削減する。

### メッセージング/イベント駆動

- **Amazon SQS**: メッセージキューでリクエストをバッファリングし、需要曲線を平坦化してピーク時の過剰プロビジョニングを防ぐ。
- **Amazon EventBridge**: イベント駆動アーキテクチャで必要なときだけ処理を実行し、ポーリングによる無駄なリソース消費を排除する。
- **AWS Step Functions**: ワークフローオーケストレーションで処理を効率的に連携させ、無駄な待機やリトライを最小化する。

### 監視・管理

- **Amazon CloudWatch**: リソース使用率を可視化し、アイドルリソースや過剰プロビジョニングを検出して削減を促す。
- **AWS Config**: 不要なリソースやコンプライアンス違反を自動検出しリソースの最適化を支援する。
- **AWS Customer Carbon Footprint Tool**: AWS利用に伴うカーボンフットプリントを可視化し、持続可能性目標への進捗を測定するダッシュボード。持続可能性の柱に直接対応するサービス。

### リージョン選択

- **リージョン選択**: 再生可能エネルギー使用率が高いリージョンを選ぶことで、ワークロード全体のカーボンフットプリントを直接削減できる。ビジネス要件と持続可能性の両方を考慮したリージョン選択が問われる。
