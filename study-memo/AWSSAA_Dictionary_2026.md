## A-Z のサービス・英字用語

### ABAC (Attribute-Based Access Control)
- **カテゴリ**: Security/IAM
- **タグ**: `#セキュリティ` `#アクセス制御` `#IAM`
- **概要**: ユーザーやリソースに付加された「属性（タグ）」に基づいてアクセス権限を決定する認可モデル。
- **SAA試験でのポイント**: 2026年試験では、タグを利用した柔軟なスケーリングが鍵。職務（Job Function）が変わってもポリ修を直接変更せず、ユーザーのタグを更新するだけで権限を管理できるメリットが問われる。
- **どういう時に使うサービスか**: 大規模組織での動的な権限管理、リソース作成者のみに操作を許可する設計。

### AWS IAM Access Analyzer
- **カテゴリ**: Security/Detection
- **タグ**: `#セキュリティ` `#監査` `#最小権限`
- **概要**: S3 バケットや IAM ロールなどが意図せず外部（パブリックや他アカウント）からアクセス可能になっていないかを自動分析するツール。
- **SAA試験でのポイント**: 「意図しない外部公開の検知」と「実際の使用履歴に基づいたポリシー生成」の 2 点が重要。最小権限の原則を実現するための強力な味方。
- **どういう時に使うサービスか**: ポリシーの公開設定確認、最小権限のポリシー作成支援。

### ALB listener rule (ALB リスナールール)
- **カテゴリ**: Networking/LoadBalancing
- **タグ**: `#ネットワーク` `#セキュリティ` `#ELB`
- **概要**: ALB が受信したトラフィックをどのターゲットグループへ転送するかを定義する条件セット。
- **SAA試験でのポイント**: 送信元 IP アドレス、HTTP ヘッダー、ホスト名、パスなどに基づく詳細な制御が可能。WAF を入れるまでもない簡易的な IP 制限や、特定のパスに対する認証（Cognito連携）のトリガーとして重要。
- **どういう時に使うサービスか**: パスベースルーティング、簡易的な IP 制限、特定のパスへの Cognito 認証適用。

### AMI (Amazon Machine Image)
- **カテゴリ**: Computing/General
- **タグ**: `#Pilot Light` `#運用上の優秀性` `#EC2`
- **概要**: EC2 インスタンスを起動するためのテンプレート。OS、ミドルウェア、アプリ設定を含めて再利用できる。
- **SAA試験でのポイント**: パイロットライトやバックアップ後の再構築では、最新のゴールデン AMI を別リージョンへコピーしておく設計が定番。EC2 Image Builder と組み合わせた自動更新も押さえる。
- **どういう時に使うサービスか**: Web サーバ群の標準化、同一構成の大量起動、DR 時の迅速な再展開。

### AWS Managed Service for Apache Flink
- **カテゴリ**: Analytics/Streaming
- **タグ**: `#分析` `#ストリーミング` `#リアルタイム`
- **概要**: ストリーミングデータを SQL や Java/Scala でリアルタイムに処理・分析するためのマネージドサービス。
- **SAA試験でのポイント**: 「収集 (Kinesis Data Streams)」と「保存/分析 (S3/OpenSearch)」の間に位置し、データがストレージに届く前にリアルタイムで集計・加工・スライディングウィンドウ処理が必要な場合の第一選択肢。Kinesis Data Analytics の後継。
- **どういう時に使うサービスか**: リアルタイム異常検知、ログのオンザフライ集計、複数のストリームの結合処理。

### Amazon API Gateway
- **カテゴリ**: AppIntegration/API
- **タグ**: `#サーバーレス` `#パフォーマンス効率` `#セキュリティ`
- **概要**: REST / HTTP / WebSocket API を公開・保護・制御するフルマネージドサービス。
- **SAA試験でのポイント**: Lambda と組み合わせたサーバーレス API、スロットリング、認証連携、キャッシュによる低レイテンシ化が頻出。WAF や Cognito と連携できる点も問われやすい。
- **どういう時に使うサービスか**: モバイルアプリの API、BFF、Webhook 受付、サーバーレスな業務 API。

### Amazon AppFlow
- **カテゴリ**: AppIntegration/DataFlow
- **タグ**: `#データ転送` `#運用上の優秀性`
- **概要**: Salesforce や Slack などの SaaS と AWS 間でデータを安全に転送するサービス。
- **SAA試験でのポイント**: SaaS 連携は AppFlow、オンプレやファイル同期は DataSync、DB 移行は SCT/DMS 系という切り分けが重要。SAA では主役より比較選択肢として出やすい。
- **どういう時に使うサービスか**: Salesforce から S3 へのデータ連携、SaaS データの定期取り込み。

### AWS Application Recovery Controller (ARC)
- **カテゴリ**: DR/Strategy
- **タグ**: `#Multi Site` `#信頼性`
- **概要**: マルチリージョン構成でフェイルオーバーの安全性を高めるためのレディネスチェックとルーティングコントロールを提供する。
- **SAA試験でのポイント**: 2026 時点でも「自動ではなく安全に切り替えたい DR」の選択肢として有効。Route 53 だけでは足りない、誤切替防止や事前検証が必要な設計で差がつく。
- **どういう時に使うサービスか**: 金融系や基幹系の手動フェイルオーバー制御、ゲームデイ訓練、複数リージョンの DR 運用。

### ARN
- **カテゴリ**: Management/Operations (概念)
- **タグ**: `#セキュリティ` `#IAM`
- **概要**: AWS リソースを一意に識別する名前。
- **SAA試験でのポイント**: IAM ポリシーの `Resource` 指定で頻出。サービスごとに形式が違うため、「どこまで絞れるか」を見る問題がある。
- **どういう時に使うサービスか**: ポリシー記述、イベントターゲット指定、リソース識別。

### AWS Artifact
- **カテゴリ**: Security/Governance
- **タグ**: `#セキュリティ` `#コンプライアンス`
- **概要**: AWS の各種コンプライアンスレポートや契約文書を取得できるサービス。
- **SAA試験でのポイント**: 「監査証跡を作る」サービスではなく、「AWS 側の監査レポートを入手する」サービス。CloudTrail や Config との混同に注意。
- **どういう時に使うサービスか**: SOC 報告書の提出、監査対応、規制業界の証跡確認。

### Amazon Athena
- **カテゴリ**: Analytics/Query
- **タグ**: `#サーバーレス` `#コスト最適化` `#パフォーマンス効率`
- **概要**: S3 上のデータに SQL を直接実行できるサーバーレス分析サービス。
- **SAA試験でのポイント**: 「保存 (S3)」されたデータを直接分析する「分析（サーバーレス分析）」の代表格。サーバー管理不要。コスト最適化のため、Glue Data Catalog との連携や、データを列指向フォーマット（Parquet/ORC）へ変換しておく設計が頻出。
- **どういう時に使うサービスか**: ログ分析、簡易データレイク分析、レポート作成前の ad-hoc クエリ。

### Amazon Aurora
- **カテゴリ**: Database/Relational
- **タグ**: `#信頼性` `#パフォーマンス効率` `#Warmup Standby`
- **概要**: MySQL / PostgreSQL 互換のマネージド RDB。分散ストレージと複数 AZ への自動耐障害設計が特徴。
- **SAA試験でのポイント**: Aurora は RDS より高可用・高性能寄り。リードレプリカ台数、フェイルオーバー、Serverless、Global Database との使い分けが重要。
- **どういう時に使うサービスか**: 高トラフィックな業務 DB、SaaS の基幹 DB、グローバル読取を伴うアプリ。

### Aurora Global Database
- **カテゴリ**: Database/Relational
- **タグ**: `#Multi Site` `#信頼性` `#マルチリージョン`
- **概要**: Aurora クラスタを複数リージョンへ拡張する機能。2026 時点の公式ドキュメントでは、プライマリ 1 リージョンに対して最大 10 個のセカンダリクラスタを持てる。
- **SAA試験でのポイント**: クロスリージョン DR の定番。RDS のクロスリージョンリードレプリカより短い RPO / RTO を狙いやすく、RDS Multi-AZ が同一リージョン HA である点との区別が必須。
- **どういう時に使うサービスか**: 多地域 DR、グローバル読取、低遅延な海外参照系。

### AWS Backup
- **カテゴリ**: Management/Backup
- **タグ**: `#Backup&Restore` `#信頼性` `#コンプライアンス`
- **概要**: 複数 AWS サービスのバックアップを一元的に定義・実行・保持管理するフルマネージドサービス。
- **SAA試験でのポイント**: 「複数サービスを一括でバックアップしたい」なら第一候補。クロスリージョン/クロスアカウントコピーと Vault Lock の組み合わせが 2026 でも重要。
- **どういう時に使うサービスか**: EBS、RDS、EFS、DynamoDB、FSx などの統合バックアップ運用。

### Amazon Bedrock
- **カテゴリ**: AI_ML/GenAI
- **タグ**: `#生成AI` `#サーバーレス` `#基盤モデル`
- **概要**: 様々な基盤モデル (FM) を API 経由で利用可能にするマネージドサービス。
- **SAA試験でのポイント**: サーバーレスで生成 AI アプリを構築する際の中心. モデルのカスタマイズ（ファインチューニングや RAG）と、データプライバシー（入力データがモデル学習に使われない等）が問われる。
- **どういう時に使うサービスか**: チャットボット、要約、画像生成などの生成 AI 機能の実装。

### AWS Budgets
- **カテゴリ**: Management/Cost
- **タグ**: `#コスト最適化`
- **概要**: 予算額を設定し、超過見込みや超過時に通知するサービス。
- **SAA試験でのポイント**: Cost Explorer は分析、Budgets はしきい値監視という役割差を問う問題が定番。コスト停止装置ではない点にも注意。
- **どういう時に使うサービスか**: 月額予算管理、部門ごとの支出アラート、PoC の費用見張り。

### AWS CDK
- **カテゴリ**: Management/IaC
- **タグ**: `#運用上の優秀性` `#IaC`
- **概要**: TypeScript や Python などでインフラを定義し、CloudFormation を生成するフレームワーク。
- **SAA試験でのポイント**: CloudFormation の上位レイヤーとして理解しておけば十分。IaC の文脈で登場しても、実体は CloudFormation ベースと整理すると迷いにくい。
- **どういう時に使うサービスか**: アプリチーム主導の IaC、再利用可能な構成部品のコード化。

### AWS Certificate Manager (ACM)
- **カテゴリ**: Security/Encryption
- **タグ**: `#セキュリティ` `#暗号化`
- **概要**: SSL/TLS 証明書の発行、配置、更新を管理するサービス。
- **SAA試験でのポイント**: ALB、CloudFront、API Gateway で HTTPS を簡単に有効化する定番。秘密鍵管理を自前で持ちたくない時の正解になりやすい。
- **どういう時に使うサービスか**: Web サイトの HTTPS 化、API の TLS 証明書管理。

### AWS CloudFormation
- **カテゴリ**: Management/IaC
- **タグ**: `#運用上の優秀性` `#IaC` `#Backup&Restore`
- **概要**: AWS リソースを YAML / JSON テンプレートで定義し、再現可能な環境を作るサービス。
- **SAA試験でのポイント**: 変更セット、ドリフト検出、スタックの再現性が基本。DR では環境再構築手段として強いが、コントロールプレーン依存である点まで理解すると精度が上がる。
- **どういう時に使うサービスか**: 標準環境の量産、開発/本番の差分削減、災害時の再作成。

### CloudFormation StackSets
- **カテゴリ**: Management/IaC
- **タグ**: `#運用上の優秀性` `#IaC` `#マルチアカウント`
- **概要**: CloudFormation スタックを複数アカウント・複数リージョンへ一括展開する機能。
- **SAA試験でのポイント**: Organizations と組み合わせた標準展開、DR リージョンへの横展開、ガバナンスの初期配布で登場しやすい。
- **どういう時に使うサービスか**: 監査用ロールの全アカウント配布、DR ベースラインの一括展開。

### Amazon CloudFront
- **カテゴリ**: Networking/CDN
- **タグ**: `#パフォーマンス効率` `#コスト最適化` `#信頼性`
- **概要**: 世界中のエッジロケーションからコンテンツを配信する CDN。
- **SAA試験でのポイント**: CloudFront は L7 配信とキャッシュ、Global Accelerator は L4 最適経路と固定 IP という差分が鉄板。オリジンフェイルオーバーや署名付き URL も出やすい。
- **どういう時に使うサービスか**: 画像/動画配信、静的サイト高速化、グローバル Web 配信。

### CloudFront + AWS WAF
- **カテゴリ**: Security/Traffic
- **タグ**: `#セキュリティ` `#Web保護` `#エッジセキュリティ`
- **概要**: CloudFront のエッジロケーションで WAF を動作させ、悪意のあるリクエストがオリジン（S3やALB）に到達する前にブロックする構成。
- **SAA試験でのポイント**: 2026 年試験では「エッジでの保護」によるオリジンの負荷軽減と防御の最大化が頻出。地理的制限（ジオブロッキング）が WAF 側でも詳細に設定できる点に注目。
- **どういう時に使うサービスか**: DDoS 対策、SQL インジェクション・XSS 防御、国別アクセス制限。

### AWS CloudHSM
- **カテゴリ**: Security/Encryption
- **タグ**: `#セキュリティ` `#コンプライアンス`
- **概要**: 専有 HSM 上で暗号鍵を管理するサービス。
- **SAA試験でのポイント**: 2026 でも「厳格な鍵管理/FIPS 要件」なら KMS ではなく CloudHSM。KMS より運用負荷が高い代わりに専有性が高い。
- **どういう時に使うサービスか**: 厳格な規制要件、専有 HSM が必要な暗号基盤。

### AWS CloudTrail
- **カテゴリ**: Security/Detection
- **タグ**: `#セキュリティ` `#監査` `#運用上の優秀性`
- **概要**: AWS API コールの監査証跡を記録するサービス。
- **SAA試験でのポイント**: 誰が何をしたかは CloudTrail、リソース状態がどう変わったかは Config。全リージョン有効化と S3 集約はほぼ定番のベストプラクティス。
- **どういう時に使うサービスか**: 操作履歴監査、不正操作の追跡、コンプライアンス証跡。

### Amazon CloudWatch
- **カテゴリ**: Management/Monitoring
- **タグ**: `#運用上の優秀性` `#信頼性` `#パフォーマンス効率`
- **概要**: メトリクス、ログ、アラーム、ダッシュボードを統合的に扱う監視サービス。
- **SAA試験でのポイント**: Auto Scaling のトリガー、SQS の深さ監視、ログ集約、メトリクスフィルターが典型。CloudWatch Logs と CloudTrail の役割差も重要。
- **どういう時に使うサービスか**: CPU アラーム、アプリログ集約、運用ダッシュボード、障害通知。

### CloudWatch Logs
- **カテゴリ**: Management/Monitoring
- **タグ**: `#運用上の優秀性` `#ログ管理`
- **概要**: CloudWatch のログ集約機能。アプリや OS、各種 AWS サービスのログを保存・検索・分析できる。
- **SAA試験でのポイント**: アプリ実行ログの保存先として頻出。CloudTrail は API 監査、CloudWatch Logs はアプリ/システムログという使い分けを明確にする。
- **どういう時に使うサービスか**: Lambda ログ、EC2 アプリログ、監視アラート用ログ収集。

### Amazon Cognito
- **カテゴリ**: Security/Identity
- **タグ**: `#セキュリティ` `#サーバーレス`
- **概要**: アプリ利用者向けの認証基盤。ユーザープールと ID プールを提供する。
- **SAA試験でのポイント**: ユーザープールは認証、ID プールは AWS リソースへの一時認証情報付与という整理が必須。外部 IdP 連携と STS も絡む。
- **どういう時に使うサービスか**: Web アプリのログイン、モバイルアプリの認証、SNS ログイン連携。

### AWS CodeBuild
- **カテゴリ**: Management/DevOps
- **タグ**: `#運用上の優秀性` `#CI/CD`
- **概要**: ソースコードのビルドとテストを実行するマネージドサービス。
- **SAA試験でのポイント**: SAA では深掘りされにくいが、CI/CD 全体像の比較問題で登場する。CodePipeline の一部として理解すれば十分。
- **どういう時に使うサービスか**: コンテナビルド、自動テスト、アーティファクト生成。

### AWS CodeDeploy
- **カテゴリ**: Management/DevOps
- **タグ**: `#運用上の優秀性` `#CI/CD`
- **概要**: EC2、Lambda、ECS へのデプロイを自動化するサービス。
- **SAA試験でのポイント**: ブルー/グリーン、ローリング、カナリアの差分を問う補助論点として出やすい。主役ではなく運用自動化の比較枠。
- **どういう時に使うサービスか**: 段階的リリース、無停止更新、自動ロールバック。

### AWS CodePipeline
- **カテゴリ**: Management/DevOps
- **タグ**: `#運用上の優秀性` `#CI/CD`
- **概要**: ソース取得からビルド、テスト、デプロイまでをつなぐパイプラインサービス。
- **SAA試験でのポイント**: 運用上の優秀性の文脈で「変更を安全に自動化する」選択肢として理解すれば十分。各ステージの連携役。
- **どういう時に使うサービスか**: 継続的デリバリー、承認付き本番反映、自動テスト付きリリース。

### AWS Compute Optimizer
- **カテゴリ**: Management/Cost
- **タグ**: `#コスト最適化` `#パフォーマンス効率`
- **概要**: 利用実績を分析して EC2、EBS、Lambda などの最適サイズを推奨するサービス。
- **SAA試験でのポイント**: Trusted Advisor は広く点検、Compute Optimizer はライトサイジングに特化という使い分けが頻出。過剰プロビジョニング対策の本命。
- **どういう時に使うサービスか**: EC2 のサイズ見直し、EBS タイプ見直し、Lambda メモリ調整。

### AWS Config
- **カテゴリ**: Security/Governance
- **タグ**: `#セキュリティ` `#コンプライアンス` `#運用上の優秀性`
- **概要**: リソース構成の変更履歴を記録し、Config ルールで準拠性を評価するサービス。
- **SAA試験でのポイント**: CloudTrail と違い、Config は構成状態の追跡が中心。非準拠検知から自動修復まで含む設計が問われやすい。
- **どういう時に使うサービスか**: 暗号化必須ルールの監視、S3 公開設定の検知、構成ドリフトの把握。

### AWS Config Rules
- **カテゴリ**: Security/Governance
- **タグ**: `#セキュリティ` `#コンプライアンス`
- **概要**: AWS リソースが特定のガイドライン（例：EBS が暗号化されているか）に準拠しているかを自動でチェックする機能。
- **SAA試験でのポイント**: 2026年試験でも「継続的なコンプライアンス監視」の主役。Organizations 全体へのルール適用や、Systems Manager Automation を使った自動修復 (Remediation) とのセットが頻出。
- **どういう時に使うサービスか**: 全 EBS の暗号化強制チェック、不要な EIP の検知、特定のタグ付与の監視。

### AWS Control Tower
- **カテゴリ**: Security/Governance
- **タグ**: `#セキュリティ` `#運用上の優秀性` `#マルチアカウント`
- **概要**: マルチアカウント環境の初期構築とガードレール適用を自動化するサービス。
- **SAA試験でのポイント**: Organizations にランディングゾーン運用を足したものとして理解すると整理しやすい。SCP や監査アカウント設計と一緒に問われる。
- **どういう時に使うサービスか**: 複数アカウントの標準化、セキュアな初期セットアップ、統制強化。

### AWS Cost Anomaly Detection
- **カテゴリ**: Management/Cost
- **タグ**: `#コスト最適化`
- **概要**: 機械学習で異常なコスト増加を検知して通知するサービス。
- **SAA試験でのポイント**: Budgets は固定しきい値、Anomaly Detection は通常パターンからの逸脱検知。急な NAT Gateway コスト増などのシナリオと相性が良い。
- **どういう時に使うサービスか**: 想定外課金の早期発見、障害や設定ミス由来の請求急増監視。

### AWS Cost and Usage Report (CUR)
- **カテゴリ**: Management/Cost
- **タグ**: `#コスト最適化`
- **概要**: 最も粒度の細かいコスト・利用データを出力するレポート。
- **SAA試験でのポイント**: 詳細分析が必要なら CUR が正解。Cost Explorer より生データ寄りという違いを押さえる。
- **どういう時に使うサービスか**: 部門別原価分析、独自 BI への連携、細粒度請求分析。

### AWS Cost Explorer
- **カテゴリ**: Management/Cost
- **タグ**: `#コスト最適化`
- **概要**: コストの可視化、傾向分析、将来予測を行うサービス。
- **SAA試験でのポイント**: 予算アラートは Budgets、詳細生データは CUR、普段の可視化は Cost Explorer という整理が鉄板。
- **どういう時に使うサービスか**: 月次コスト確認、サービス別コスト分析、節約候補の把握。

### Amazon Data Firehose
- **カテゴリ**: Analytics/Streaming
- **タグ**: `#パフォーマンス効率` `#運用上の優秀性`
- **概要**: ストリーミングデータを自動バッファリングしながら S3、Redshift、OpenSearch などへ配信するサービス。
- **SAA試験でのポイント**: 「収集・配信」レイヤー。データを「保存 (S3)」や「分析 (Redshift/OpenSearch)」へニアリアルタイムで自動搬送する。マネージドで自動スケーリングされるため Firehose は運用負荷が低い。
- **どういう時に使うサービスか**: ログ集約、クリックストリーム転送、監査ログの S3 保存。

### Amazon Data Lifecycle Manager (DLM)
- **カテゴリ**: Management/Backup
- **タグ**: `#Backup&Restore` `#運用上の優秀性`
- **概要**: EBS スナップショットや AMI の作成・保持・削除を自動化するサービス。
- **SAA試験でのポイント**: EBS の世代管理をシンプルに自動化したい場合の正解。フルバックアップではなく増分スナップショット前提で出る。
- **どういう時に使うサービスか**: 日次スナップショット、AMI 世代管理、保管期間の自動化。

### AWS Database Migration Service (DMS)
- **カテゴリ**: Migration/Database
- **タグ**: `#移行` `#データベース` `#信頼性`
- **概要**: データベースを安全かつ最小限のダウンタイムで AWS へ移行するためのサービス。
- **SAA試験でのポイント**: 同種 DB 移行だけでなく、異種 DB 移行（SCT との組み合わせ）が頻出。継続的なデータレプリケーション機能により、ダウンタイムを最小化できる点が最大のメリット。
- **どういう時に使うサービスか**: オンプレ DB の AWS 移行、Aurora への継続的同期、異種エンジンの移行。

### AWS DataSync
- **カテゴリ**: Migration/Data
- **タグ**: `#データ転送` `#移行`
- **概要**: オンプレミスや他クラウドから AWS ストレージへ高速転送・同期するサービス。
- **SAA試験でのポイント**: ネットワーク経由の大量ファイル同期は DataSync、物理搬送は Snow Family、ハイブリッド常設は Storage Gateway という切り分けが重要。
- **どういう時に使うサービスか**: NAS から S3/EFS/FSx への移行、定期同期、バックアップ転送。

### Amazon Detective
- **カテゴリ**: Security/Detection
- **タグ**: `#セキュリティ` `#調査`
- **概要**: GuardDuty などの検出を起点に、関連するセキュリティイベントの因果関係を可視化するサービス。
- **SAA試験でのポイント**: 検知は GuardDuty、調査は Detective、集約は Security Hub と整理すると混乱しない。
- **どういう時に使うサービスか**: インシデント調査、侵害経路の可視化、原因分析。

### AWS Direct Connect
- **カテゴリ**: Networking/Connectivity
- **タグ**: `#ネットワーク` `#パフォーマンス効率`
- **概要**: オンプレミスと AWS を専用線で接続するサービス。
- **SAA試験でのポイント**: Site-to-Site VPN より安定・低遅延・高帯域。高価だが本番基幹接続で選ばれやすい。冗長化設計もよく問われる。
- **どういう時に使うサービスか**: 基幹システム接続、大容量転送、安定したハイブリッド接続。

### AWS Directory Service
- **カテゴリ**: Security/IAM
- **タグ**: `#セキュリティ` `#ハイブリッド`
- **概要**: Active Directory と AWS を連携させるマネージドディレクトリサービス。
- **SAA試験でのポイント**: AWS Managed Microsoft AD、AD Connector、Simple AD の違いが重要。Windows ベースの認証統合問題でよく出る。
- **どういう時に使うサービスか**: AD 認証のクラウド延伸、FSx for Windows 連携、社内 Windows 認証統合。

### Amazon DocumentDB
- **カテゴリ**: Database/NoSQL
- **タグ**: `#パフォーマンス効率` `#移行`
- **概要**: MongoDB 互換 API を持つマネージドドキュメントデータベース。
- **SAA試験でのポイント**: 「MongoDB 互換のマネージドサービス」がキーワード。 DynamoDB とはデータモデルが違うため、JSON 風ドキュメント互換性を求める場面で選ぶ。
- **どういう時に使うサービスか**: MongoDB ワークロードの AWS 移行、ドキュメント中心アプリ。

### Amazon DynamoDB
- **カテゴリ**: Database/NoSQL
- **タグ**: `#信頼性` `#パフォーマンス効率` `#サーバーレス`
- **概要**: ミリ秒台の低レイテンシで動作するフルマネージド NoSQL データベース。
- **SAA試験でのポイント**: オンデマンド/プロビジョンド、PITR、Streams、DAX、Global Tables の組み合わせが頻出。高スケール・マルチ AZ は前提知識として扱う。
- **どういう時に使うサービスか**: セッション、ゲームランキング、IoT 状態管理、高スループットなキー値アクセス。

### DynamoDB Accelerator (DAX)
- **カテゴリ**: Database/NoSQL
- **タグ**: `#パフォーマンス効率` `#キャッシュ`
- **概要**: DynamoDB 向けのインメモリキャッシュ。読み取りレイテンシをさらに短縮する。
- **SAA試験でのポイント**: マイクロ秒単位の読取高速化が必要な時に有効。ただし強整合読み取りが必要な要件とは相性が悪い。
- **どういう時に使うサービスか**: 高速ランキング、入札、瞬間的な参照負荷が高い読取中心ワークロード。

### DynamoDB Global Tables
- **カテゴリ**: Database/NoSQL
- **タグ**: `#Multi Site` `#信頼性` `#マルチリージョン`
- **概要**: DynamoDB のマルチリージョン・マルチアクティブ複製機能。2026 時点の公式では MREC と MRSC の整合性モードがある。
- **SAA試験でのポイント**: NoSQL の多地域 DR では第一候補。各リージョンで読み書きでき、昇格操作が不要という点が Aurora Global Database と大きく異なる。
- **どういう時に使うサービスか**: グローバルサービス、リージョン障害に強いユーザープロファイル、世界分散アプリ。

### DynamoDB Streams
- **カテゴリ**: AppIntegration/Workflow
- **タグ**: `#サーバーレス` `#運用上の優秀性`
- **概要**: DynamoDB テーブルの変更履歴を時系列で保持し、Lambda などへ連携できる機能。
- **SAA試験でのポイント**: 変更イベントのリアクティブ処理に使う。監査用の CloudTrail とは目的が違う点が問われやすい。
- **どういう時に使うサービスか**: 更新イベント通知、派生テーブル更新、非同期連携。

### Amazon EBS
- **カテゴリ**: Storage/Block
- **タグ**: `#パフォーマンス効率` `#信頼性` `#EC2`
- **概要**: EC2 向けのブロックストレージ。単一 AZ に属するが、AZ 内で高耐久に設計されている。
- **SAA試験でのポイント**: gp3 と io2 の使い分け、EBS は複数 EC2 共有向きではないこと、スナップショットが増分バックアップであることが基本。
- **どういう時に使うサービスか**: OS ディスク、DB ディスク、低レイテンシなブロックストレージ。

### EBS Snapshot
- **カテゴリ**: Management/Backup
- **タグ**: `#Backup&Restore` `#EBS` `#バックアップ`
- **概要**: EBS ボリュームのバックアップ。初回フル、以降増分で保存される。
- **SAA試験でのポイント**: クロスリージョンコピー、AMI 作成の元、DLM 自動化が典型論点。ボリューム障害対策と DR の両方で登場する。
- **どういう時に使うサービスか**: EC2 のバックアップ、DR 用コピー、定期復元ポイント作成。

### EBS / RDS / S3 暗号化
- **カテゴリ**: Security/Encryption
- **タグ**: `#セキュリティ` `#暗号化` `#KMS`
- **概要**: AWS の各主要ストレージサービスで KMS (AWSマネージド型またはカスタマー管理型) を利用して保存データを暗号化する機能。
- **SAA試験でのポイント**: 2026 年時点では「デフォルトでの暗号化」および「KMS キーの使い分け（AWS管理/カスタマー管理/BYOK）」が重要な判断基準。暗号化されていないスナップショットを暗号化するには、一度コピーが必要になる点は試験の定番。
- **どういう時に使うサービスか**: 保存データの保護要件、規制準拠、マルチアカウント間での安全なデータ共有。

### Amazon EC2
- **カテゴリ**: Computing/General
- **タグ**: `#信頼性` `#パフォーマンス効率` `#コスト最適化`
- **概要**: AWS 上の仮想サーバー。インスタンスタイプ、購入オプション、AZ 配置を柔軟に選べる。
- **SAA試験でのポイント**: Auto Scaling、ELB、EBS、IAM ロールとの組み合わせが基礎。要件からインスタンスファミリーや購入モデルを選ぶ問題が多い。
- **どういう時に使うサービスか**: Web サーバ、業務アプリ、バッチ、レガシー移行先。

### EC2 Auto Scaling
- **カテゴリ**: Computing/General
- **タグ**: `#信頼性` `#コスト最適化` `#Warmup Standby`
- **概要**: EC2 台数を需要に応じて自動増減する機能。
- **SAA試験でのポイント**: CloudWatch アラーム連携、ターゲット追跡、スケジュール、最小容量 0 のパイロットライト設計が頻出。DR と通常運用の両方で出る。
- **どういう時に使うサービスか**: 負荷変動のある Web サーバ、ウォームスタンバイ環境、夜間停止を伴う開発環境。

### EC2 Spot Instances
- **カテゴリ**: Computing/General
- **タグ**: `#コスト最適化` `#EC2`
- **概要**: 余剰キャパシティを大幅割引で利用できる EC2 の購入オプション。
- **SAA試験でのポイント**: 中断されてもよいバッチや CI/CD に向く。 2 分前通知がある一方、停止耐性のない本番 DB には不向きという判断が重要。
- **どういう時に使うサービスか**: バッチ処理、レンダリング、検証環境、分散ワーカー。

### Amazon ECS
- **カテゴリ**: Computing/Container
- **タグ**: `#運用上の優秀性` `#パフォーマンス効率`
- **概要**: AWS ネイティブなコンテナオーケストレーションサービス。
- **SAA試験でのポイント**: EKS より学習コストが低く、Fargate と組み合わせやすい。 `taskRoleArn` と `executionRoleArn` の違いも周辺知識として有効。
- **どういう時に使うサービスか**: マイクロサービス、バッチコンテナ、AWS 中心のコンテナ基盤。

### Amazon EFS
- **カテゴリ**: Storage/File
- **タグ**: `#信頼性` `#パフォーマンス効率` `#共有ストレージ`
- **概要**: マルチ AZ で利用できる NFS ベースのフルマネージド共有ファイルストレージ。
- **SAA試験でのポイント**: EBS と違って複数 EC2 から同時マウントできる。2026 時点の公式では、レプリケーション後の RPO は多くのファイルシステムで 15 分が目安。
- **どういう時に使うサービスか**: 共有アップロード領域、CMS、複数 Linux サーバからの共通ファイル参照。

### Amazon EKS
- **カテゴリ**: Computing/Container
- **タグ**: `#運用上の優秀性` `#パフォーマンス効率`
- **概要**: マネージド Kubernetes サービス。
- **SAA試験でのポイント**: HPA が Pod 数、Cluster Autoscaler がノード数を増減する役割分担が重要。ECS と EKS の比較問題で出やすい。
- **どういう時に使うサービスか**: Kubernetes ベースの既存資産活用、コンテナ標準化、マルチクラウド志向の基盤。

### Amazon ElastiCache
- **カテゴリ**: Database/InMemory
- **タグ**: `#パフォーマンス効率` `#信頼性` `#キャッシュ`
- **概要**: Redis / Memcached ベースのマネージドインメモリキャッシュ。
- **SAA試験でのポイント**: Redis はレプリカ・フェイルオーバー・バックアップあり、Memcached はシンプルなキャッシュ専用。キャッシュ戦略もあわせて問われる。
- **どういう時に使うサービスか**: セッション管理、商品一覧キャッシュ、DB 負荷軽減。

### ElastiCache Global Datastore
- **カテゴリ**: Database/InMemory
- **タグ**: `#Multi Site` `#信頼性` `#マルチリージョン`
- **概要**: ElastiCache for Redis のクロスリージョン複製機能。
- **SAA試験でのポイント**: キャッシュ層やセッション層の DR で登場する。DB 本体の DR とは役割が違うため、Aurora / DynamoDB と混同しないこと。
- **どういう時に使うサービスか**: グローバルセッション管理、リージョン障害時のキャッシュ継続。

### AWS Elastic Beanstalk
- **カテゴリ**: Computing/General
- **タグ**: `#運用上の優秀性`
- **概要**: アプリケーションコードを渡すと、EC2、ELB、Auto Scaling などをまとめて構成してくれる PaaS。
- **SAA試験でのポイント**: 「インフラ詳細よりデプロイ簡便性を優先」する時の選択肢。 Lambda ほどサーバーレスではない点が比較ポイント。
- **どういう時に使うサービスか**: 小〜中規模 Web アプリの迅速立ち上げ、既存コードの簡単デプロイ。

### AWS Elastic Disaster Recovery (DRS)
- **カテゴリ**: Migration/Server
- **タグ**: `#Pilot Light` `#信頼性` `#移行`
- **概要**: サーバーのブロックレベル継続レプリケーションで AWS への DR を実現するサービス。
- **SAA試験でのポイント**: オンプレ/他クラウドから AWS への DR で頻出。旧 CloudEndure 系の後継として、低コストなステージング + フェイルオーバー時拡張という考え方が重要。
- **どういう時に使うサービスか**: サーバ単位の DR、データセンター移行、災害時の迅速起動。

### Elastic IP
- **カテゴリ**: Networking/Access
- **タグ**: `#ネットワーク` `#コスト最適化`
- **概要**: AWS 上で固定的に使えるパブリック IPv4 アドレス。
- **SAA試験でのポイント**: 使っていない Elastic IP に課金が発生する点は定番。高可用化手段そのものではなく、アドレス固定用途。
- **どういう時に使うサービスか**: 固定送信元 IP、NAT インスタンス、移行時の一時的なアドレス固定。

### Elastic Load Balancing (ELB)
- **カテゴリ**: Networking/LoadBalancing
- **タグ**: `#信頼性` `#パフォーマンス効率`
- **概要**: 複数ターゲットへトラフィックを分散するロードバランサ群の総称。
- **SAA試験でのポイント**: ALB は HTTP/HTTPS の L7、NLB は TCP/UDP の L4、Gateway Load Balancer は仮想アプライアンス向け。ヘルスチェックとマルチ AZ が基本論点。
- **どういう時に使うサービスか**: Web 負荷分散、マイクロサービス入口、低遅延 TCP/UDP 分散。

### Amazon EMR
- **カテゴリ**: Analytics/BigData
- **タグ**: `#パフォーマンス効率` `#ビッグデータ`
- **概要**: Hadoop、Spark などを使う大規模データ処理基盤。
- **SAA試験でのポイント**: 極めて大規模な「ETL・分析」を、Spark や Hadoop を使用してカスタムに行いたい場合の選択肢。Athena より自由度が高いが EC2 サーバー管理が必要。
- **どういう時に使うサービスか**: バッチ分析、Spark ジョブ、データレイク変換基盤。

### Amazon EventBridge
- **カテゴリ**: AppIntegration/Workflow
- **タグ**: `#サーバーレス` `#運用上の優秀性`
- **概要**: AWS サービスや SaaS、独自アプリのイベントをルールベースでルーティングするサービス。
- **SAA試験でのポイント**: ポーリングを避けるイベント駆動設計の中心。 CloudWatch Events の後継として理解しておけばよい。
- **どういう時に使うサービスか**: セキュリティ検知から自動修復、スケジュール実行、イベント連携。

### AWS Fargate
- **カテゴリ**: Computing/Container
- **タグ**: `#サーバーレス` `#コスト最適化` `#運用上の優秀性`
- **概要**: ECS / EKS で利用できるサーバーレスコンピュート実行基盤。
- **SAA試験でのポイント**: EC2 管理不要で vCPU / メモリ単位課金。可変負荷でコンテナを簡単運用したい場面の正解になりやすい。
- **どういう時に使うサービスか**: 小規模コンテナ基盤、バッチコンテナ、運用負荷を抑えたマイクロサービス。

### AWS Firewall Manager
- **カテゴリ**: Security/Traffic
- **タグ**: `#セキュリティ` `#ガバナンス`
- **概要**: 複数アカウントにまたがる WAF、Shield Advanced、Security Group などのポリシーを一元管理するサービス。
- **SAA試験でのポイント**: トラフィック検査そのものをするのは WAF / Network Firewall。Firewall Manager はその統制レイヤー。
- **どういう時に使うサービスか**: Organizations 全体でのセキュリティルール標準化。

### Amazon FSx
- **カテゴリ**: Storage/File
- **タグ**: `#共有ストレージ` `#Windows` `#ハイブリッド`
- **概要**: Windows File Server、NetApp ONTAP、Lustre など用途別ファイルシステムを提供するサービス群。
- **SAA試験でのポイント**: SMB と AD が必要なら FSx for Windows、NFS 高性能/HPC なら FSx for Lustre、企業ファイル機能なら ONTAP と整理すると選びやすい。
- **どういう時に使うサービスか**: Windows ファイルサーバ移行、HPC、エンタープライズ NAS。

### AWS Global Accelerator
- **カテゴリ**: Networking/Access
- **タグ**: `#パフォーマンス効率` `#信頼性` `#マルチリージョン`
- **概要**: Anycast の固定 IP と AWS グローバルネットワークを使い、最適なリージョンやエンドポイントへトラフィックを導くサービス。
- **SAA試験でのポイント**: CloudFront と違いキャッシュではなく通信経路最適化。DNS TTL に依存しない高速フェイルオーバーも重要。
- **どういう時に使うサービスか**: グローバルな API、ゲーム、金融系トラフィック、高速フェイルオーバー。

### AWS Glue
- **カテゴリ**: Analytics/Catalog
- **タグ**: `#運用上の優秀性` `#ETL`
- **概要**: データ検出、変換、カタログ化を行うサーバーレス ETL サービス。
- **SAA試験でのポイント**: 「ETL」レイヤーの代表。データを「保存（S3）」する前に、分析に適した形式 (Parquet/ORC等) へ変換し、「ガバナンス (Glue Data Catalog)」を作成する役割を担う。
- **どういう時に使うサービスか**: S3 データ整形、ログ ETL、データレイクのカタログ管理。

### AWS Glue Data Catalog
- **カテゴリ**: Analytics/Catalog
- **タグ**: `#分析` `#メタデータ` `#ガバナンス`
- **概要**: データの構造（スキーマ）や保存場所を管理する、集中管理型の中央メタデータリポジトリ。
- **SAA試験でのポイント**: 「ガバナンス・メタデータ管理」レイヤーの中心。Athena、Redshift Spectrum、EMR が S3 上のデータをテーブルとして認識するための「共通の辞書」として機能する。
- **どういう時に使うサービスか**: データレイク上のテーブル定義管理、スキーマ情報の共有、クローラによる自動メタデータ収集。

### AWS Graviton
- **カテゴリ**: Computing/General
- **タグ**: `#コスト最適化` `#パフォーマンス効率` `#持続可能性`
- **概要**: AWS 独自の ARM ベースプロセッサ。
- **SAA試験でのポイント**: 2026 でも「コスト性能比改善」の代表例。アーキテクチャ互換性に問題がなければ、EC2 や RDS の節約策として出やすい。
- **どういう時に使うサービスか**: Linux 系 Web アプリ、コンテナ、コスト重視の一般計算。

### Amazon GuardDuty
- **カテゴリ**: Security/Detection
- **タグ**: `#セキュリティ` `#検知`
- **概要**: AWS ログをもとに脅威を自動検出するマネージドサービス。
- **SAA試験でのポイント**: 有効化が簡単で「まず検知したい」時の定番。WAF や Shield のようにブロック主体ではなく検知主体である点が重要。
- **どういう時に使うサービスか**: 不審 API、異常 DNS、侵害兆候の可視化。

### AWS Health Dashboard
- **カテゴリ**: Management/Operations
- **タグ**: `#運用上の優秀性` `#監視`
- **概要**: AWS 側の障害・メンテナンス・影響イベントを通知するサービス。
- **SAA試験でのポイント**: 自社アプリ監視の CloudWatch とは別物。AWS 側起因のイベントを知りたい場面で正解になる。
- **どういう時に使うサービスか**: メンテナンス通知確認、障害影響の把握、運用周知。

### IAM
- **カテゴリ**: Security/IAM
- **タグ**: `#セキュリティ` `#アクセス制御`
- **概要**: AWS リソースへのアクセス権限をユーザー、ロール、ポリシーで制御する基盤サービス。
- **SAA試験でのポイント**: デフォルト拒否、明示的 Deny 優先、ロール利用、クロスアカウント AssumeRole が最重要。キーワードは最小権限。

### IAM Identity Center

- **タグ**: `#セキュリティ` `#マルチアカウント`
- **概要**: AWS アカウントやアプリへのシングルサインオンを一元管理するサービス。
- **SAA試験でのポイント**: Organizations と組み合わせたマルチアカウント運用の標準選択肢。SAML フェデレーションとも結び付けて理解する。
- **どういう時に使うサービスか**: 企業 SSO、複数 AWS アカウントへの統合ログイン、権限セット管理。

### Identity Pool (ID プール)

- **タグ**: `#Cognito` `#STS`
- **概要**: Cognito で認証済み/未認証ユーザーへ AWS 一時認証情報を払い出す機能。
- **SAA試験でのポイント**: ユーザープールで認証し、ID プールで AWS リソースアクセス権を与える流れが典型。
- **どういう時に使うサービスか**: アプリ利用者に S3 直接アップロードを許可する時など。

### Amazon Inspector
- **カテゴリ**: Security/Detection
- **タグ**: `#セキュリティ` `#脆弱性管理`
- **概要**: EC2 やコンテナイメージの脆弱性を検出するサービス。
- **SAA試験でのポイント**: 構成監査は Config、脅威検知は GuardDuty、脆弱性検出は Inspector と役割を区別する。
- **どういう時に使うサービスか**: OS パッケージ脆弱性確認、イメージスキャン、継続的な脆弱性管理。

### Amazon Kinesis Data Streams
- **カテゴリ**: Analytics/Streaming
- **タグ**: `#パフォーマンス効率` `#リアルタイム`
- **概要**: リアルタイムデータストリームを取り込み、自前のコンシューマで処理できるサービス。
- **SAA試験でのポイント**: 「収集」レイヤー。数ミリ秒のリアルタイム性が求められる独自処理や、Apache Flink (分析) へのデータ供給口として使用する。シャードによるプロビジョニングが必要な点に注意。
- **どういう時に使うサービスか**: センサーデータ、アクセスログの即時処理、不正検知。

### AWS KMS
- **カテゴリ**: Security/Encryption
- **タグ**: `#セキュリティ` `#暗号化`
- **概要**: 暗号鍵を作成・保管・利用制御するマネージド KMS。
- **SAA試験でのポイント**: キーポリシーと IAM の両方が絡むこと、S3/EBS/RDS/Secrets Manager との統合が基本。CloudHSM との差も重要。
- **どういう時に使うサービスか**: 保存データ暗号化、アプリ秘密鍵管理、監査付き鍵利用。

### KMS Key Policy (キーポリシー)
- **カテゴリ**: Security/IAM

- **概要**: KMS キーに対するアクセス制御を定義するリソースベースポリシー。
- **SAA試験でのポイント**: KMS キーを利用するには、IAM ポリシーだけでなく、キーポリシー側での許可も必須。デフォルトでは「ルートアカウント」に全権限を委譲する設定が含まれる。クロスアカウントでのキー利用時は、双方（相手のIAMポリとこちらのキーポリ）の許可が必要。
- **どういう時に使うサービスか**: KMS キーの細粒度アクセス制御、クロスアカウントでの暗号化リソース共有。

### AWS Lake Formation
- **カテゴリ**: Analytics/Catalog
- **タグ**: `#データレイク` `#セキュリティ` `#ガバナンス`
- **概要**: データレイクの構築とアクセス制御を簡素化するサービス。
- **SAA試験でのポイント**: 「ガバナンス」レイヤーにおいて、S3 等のデータレイクに対するアクセス権限（データベース、テーブル、カラム、行レベル）を一元管理する。Glue Data Catalog の権限管理をさらに強化（精密化）する役割を持つ。

### AWS Lambda

- **SAA試験でのポイント**: API Gateway、S3、EventBridge、DynamoDB Streams との連携が頻出。短時間処理、アイドル課金なし、運用負荷低減がキーワード。
- **どういう時に使うサービスか**: 画像変換、通知処理、イベント駆動自動化、軽量 API バックエンド。

### Amazon Macie

- **概要**: S3 内の機密データを検出・可視化するサービス。
- **SAA試験でのポイント**: 「S3 の個人情報や機微情報を探したい」なら Macie。WAF や GuardDuty と用途が全く違う。
- **どういう時に使うサービスか**: PII 検出、機密データ棚卸し、S3 の情報漏えい対策。

### NAT Gateway

- **SAA試験でのポイント**: 高可用だがデータ処理課金が重くなりやすい。S3/DynamoDB 向けは VPC エンドポイントで代替するとコストを抑えやすい。
- **どういう時に使うサービスか**: パッチ取得、外部 API 呼び出し、私設サブネットからの外向き通信。

### Amazon MQ

- **SAA試験でのポイント**: レガシーメッセージブローカー移行なら MQ、AWS ネイティブな疎結合なら SQS/SNS と判断する。
- **どういう時に使うサービスか**: 既存 JMS ベースシステムの移行、RabbitMQ 互換運用。

### AWS Network Firewall

- **タグ**: `#セキュリティ` `#ネットワーク`
- **概要**: VPC レベルでトラフィックを検査するマネージドファイアウォール。
- **SAA試験でのポイント**: SG/NACL より高度な検査、WAF より下位レイヤの制御。どこに置くか、何を見たいかで選ぶ。
- **どういう時に使うサービスか**: 東西/南北トラフィック検査、IDS/IPS 的な制御、VPC 境界保護。

### Amazon OpenSearch Service
- **カテゴリ**: Analytics/Query
- **タグ**: `#パフォーマンス効率` `#検索` `#ログ分析`
- **概要**: ログ分析や全文検索向けのマネージド検索サービス。
- **SAA試験でのポイント**: 「分析（検索/ログ分析）」レイヤー。全文検索や可視化 (OpenSearch Dashboards) を伴うトラブルシューティングが得意。「収集（Firehose）」から直接データが届く構成が定番。
- **どういう時に使うサービスか**: アプリ全文検索、運用ログ分析、可観測性基盤。

### AWS Organizations

- **タグ**: `#セキュリティ` `#コスト最適化` `#マルチアカウント`
- **概要**: 複数 AWS アカウントを組織単位で管理するサービス。
- **SAA試験でのポイント**: SCP、OU、一括請求、RI/Savings Plans 共有が主要論点。Identity Center や Control Tower とセットで問われる。
- **どういう時に使うサービスか**: 開発/本番分離、部門別アカウント管理、統制強化。

### Systems Manager Parameter Store
- **カテゴリ**: Security/Encryption
- **タグ**: `#セキュリティ` `#運用上の優秀性`
- **概要**: アプリ設定値やシークレットを階層構造で保存できる機能。
- **SAA試験でのポイント**: Secrets Manager との違いは、自動ローテーションや DB 認証情報管理の強さ。コストを抑えた設定管理では Parameter Store が有力。
- **どういう時に使うサービスか**: アプリ設定、簡易シークレット保存、環境変数集中管理。

### AWS Pricing Calculator

- **タグ**: `#コスト最適化`
- **概要**: AWS 構成の概算費用を見積もるツール。
- **SAA試験でのポイント**: 実績分析は Cost Explorer、事前見積もりは Pricing Calculator。移行前の概算で選ばせる問題に出る。
- **どういう時に使うサービスか**: 導入前見積もり、提案書作成、構成比較。

### AWS PrivateLink
- **カテゴリ**: Networking/Access
- **タグ**: `#セキュリティ` `#ネットワーク`
- **概要**: インターフェイス VPC エンドポイント経由で AWS サービスや独自サービスへ私設接続する仕組み。
- **SAA試験でのポイント**: PrivateLink はインターフェイス型 VPC エンドポイントの基盤。S3/DynamoDB の無料ゲートウェイ型とは別物。
- **どういう時に使うサービスか**: SaaS の私設公開、インターネット非経由のサービス接続、共有サービス基盤。

### Amazon QuickSight
- **カテゴリ**: Analytics/Visualization
- **タグ**: `#分析` `#可視化`
- **概要**: AWS データソースを可視化する BI サービス。
- **SAA試験でのポイント**: 「可視化」レイヤー。Athena や Redshift と連携し、分析結果をグラフやダッシュボード化する。SPICE エンジンの活用による高速な応答性能も試験ポイント。
- **どういう時に使うサービスか**: KPI ダッシュボード、分析結果の共有、運用可視化。

### Resource-based Policy (リソースベースポリシー)
- **カテゴリ**: Security/ResourcePolicy
- **タグ**: `#セキュリティ` `#IAM`
- **概要**: S3 バケット、SQS キュー、KMS キーなど、特定のリソースに直接アタッチするポリシー。
- **SAA試験でのポイント**: IAM ポリシー (Identity-based) との違いを明確にする。リソースに対して「誰が」アクセスできるかを「リソース側」で定義する。クロスアカウントアクセスを許可したい場合、多くの場合でリソースベースポリシーが必要になる。

### Reserved Instances (RI)
- **カテゴリ**: Management/Cost
- **タグ**: `#コスト最適化`
- **概要**: 1 年または 3 年の利用コミットにより割引を受ける購入オプション。
- **SAA試験でのポイント**: 安定稼働する EC2 / RDS / ElastiCache / Redshift に有効。Savings Plans より柔軟性が低い代わりに、条件次第で有利。
- **どういう時に使うサービスか**: 長期間止まらない本番 DB、定常負荷のアプリサーバ。

### Amazon RDS
- **カテゴリ**: Database/Relational
- **タグ**: `#信頼性` `#Warmup Standby` `#パフォーマンス効率`
- **概要**: MySQL、PostgreSQL などをマネージドで運用できるリレーショナルデータベースサービス。
- **SAA試験でのポイント**: Multi-AZ は HA、リードレプリカはスケール/DR 補助、Aurora とは特性が違う。自動バックアップと PITR も頻出。
- **どういう時に使うサービスか**: 業務トランザクション DB、既存 RDB 移行、一般的な Web アプリ DB。

### Amazon RDS Multi-AZ DB Cluster
- **カテゴリ**: Database/Relational
- **タグ**: `#信頼性` `#高可用性`
- **概要**: 同一リージョン 3 AZ に writer 1 台と readable standby 2 台を配置する RDS の高可用性構成。
- **SAA試験でのポイント**: 2026 時点の公式でも「同一リージョンの HA」であり、クロスリージョン DR ではない。読取容量も増やせるが、用途の本質は可用性向上。
- **どういう時に使うサービスか**: AZ 障害対策、低停止時間の業務 DB、同一リージョン内の高可用構成。

### Amazon RDS Proxy
- **カテゴリ**: Database/Relational
- **タグ**: `#パフォーマンス効率` `#信頼性`
- **概要**: RDS / Aurora への接続をプールし、接続集中を緩和するプロキシ。
- **SAA試験でのポイント**: 2026 時点の公式でもマルチ AZ に高可用展開される。Lambda など短命接続が多いアプリの DB 接続爆発対策として出やすい。
- **どういう時に使うサービスか**: Lambda からの大量接続、接続数上限対策、フェイルオーバー時の接続安定化。

### Amazon Redshift
- **カテゴリ**: Database/Warehouse
- **タグ**: `#パフォーマンス効率` `#データウェアハウス` `#BI`
- **概要**: 列指向・MPP ベースのデータウェアハウスサービス。
- **SAA試験でのポイント**: 「分析（データウェアハウス）」レイヤー。大規模・複雑な集計が得意。BI ツール (QuickSight) のバックエンドとして最適。S3 上のデータも Redshift Spectrum で直接クエリ可能。
- **どういう時に使うサービスか**: BI 集計、経営分析、データマート。

### Amazon Route 53
- **カテゴリ**: Networking/DNS
- **タグ**: `#信頼性` `#Multi Site` `#グローバル`
- **概要**: DNS、ドメイン登録、ヘルスチェック、ルーティングポリシーを提供するサービス。
- **SAA試験でのポイント**: フェイルオーバー、レイテンシ、加重、地理的ルーティングを要件から選べるようにする。DR 問題の登場頻度が非常に高い。
- **どういう時に使うサービスか**: DNS 管理、リージョン切替、Blue/Green 流量制御。

### RPO
- **カテゴリ**: DR/Metrics
- **タグ**: `#Backup&Restore`
- **概要**: どこまでのデータ損失を許容するかを表す目標値。
- **SAA試験でのポイント**: RTO とセットで必ず問われる。 Aurora Global Database、CRR、Backup 戦略との対応付けが重要。
- **どういう時に使うサービスか**: DR 戦略選定、バックアップ設計、業務要件整理。

### RTO
- **カテゴリ**: DR/Metrics
- **タグ**: `#Backup&Restore`
- **概要**: 復旧までに許容できる停止時間を表す目標値。
- **SAA試験でのポイント**: マルチサイトが最短、バックアップ&リストアが最長という基本整理が重要。設問の要求から DR 戦略を選ばせる軸になる。
- **どういう時に使うサービスか**: DR 戦略選定、復旧手順設計、SLA 定義。

### Amazon S3
- **カテゴリ**: Storage/Object
- **タグ**: `#信頼性` `#コスト最適化` `#Backup&Restore`
- **概要**: 高耐久なオブジェクトストレージ。多様なストレージクラスとライフサイクル機能を持つ。
- **SAA試験でのポイント**: 分析レイヤーにおける「保存（データレイク）」の役割を担う。分析効率化のため、Glue 等を用いて JSON を Parquet/ORC 等の列指向フォーマットに変換して保存するのがベストプラクティス。
- **どういう時に使うサービスか**: バックアップ、ログ保存、静的コンテンツ配信、データレイク。

### S3 Cross-Region Replication (CRR)
- **カテゴリ**: Storage/Object
- **タグ**: `#Multi Site` `#マルチリージョン`
- **概要**: S3 オブジェクトを別リージョンのバケットへ自動複製する機能。
- **SAA試験でのポイント**: バージョニングが必須。2026 時点でも S3 Replication Time Control により 15 分以内複製の SLA を伴う設計が重要論点。
- **どういう時に使うサービスか**: 静的アセット DR、ログの別リージョン退避、コンプライアンス複製。

### S3 Intelligent-Tiering
- **カテゴリ**: Storage/Object
- **タグ**: `#コスト最適化` `#ストレージ`
- **概要**: アクセスパターンに応じてオブジェクトを自動でコスト効率のよい層へ移すストレージクラス。
- **SAA試験でのポイント**: 2026 時点の公式では 30 日未アクセスで IA、90 日で Archive Instant Access などへ自動遷移。アクセス予測が難しいデータの定番解。
- **どういう時に使うサービスか**: アクセス頻度が読めないログ、データレイク、長期保管データ。

### Amazon S3 Transfer Acceleration
- **カテゴリ**: Migration/Data
- **タグ**: `#パフォーマンス効率` `#グローバル`
- **概要**: CloudFront のエッジネットワークを使い、遠距離からの S3 アップロードを高速化する機能。
- **SAA試験でのポイント**: 世界中の拠点から S3 へ高速アップロードしたい場面で有効。配信高速化の CloudFront と用途が逆向き。
- **どういう時に使うサービスか**: 海外拠点からの大容量アップロード、グローバルなデータ収集。

### Savings Plans
- **カテゴリ**: Management/Cost
- **タグ**: `#コスト最適化`
- **概要**: 一定の利用コミットメントに対して幅広いコンピュート料金を割引する料金モデル。
- **SAA試験でのポイント**: 2026 でも RI より柔軟な節約策として重要。 Compute Savings Plans と Instance Savings Plans の柔軟性差を押さえる。
- **どういう時に使うサービスか**: 長期運用する EC2/Fargate/Lambda のコスト削減、変動のある本番基盤。

### AWS SAM
- **カテゴリ**: Management/IaC
- **タグ**: `#サーバーレス` `#IaC`
- **概要**: サーバーレスアプリを簡潔に定義・デプロイする CloudFormation 拡張。
- **SAA試験でのポイント**: Lambda、API Gateway、DynamoDB などの構成を簡単に表す枠組みとして理解しておけば十分。
- **どういう時に使うサービスか**: サーバーレス API の IaC、開発用テンプレート管理。

### AWS Schema Conversion Tool (SCT)
- **カテゴリ**: Migration/Database
- **タグ**: `#移行` `#データベース`
- **概要**: 異種 DB 間のスキーマ変換を支援するツール。
- **SAA試験でのポイント**: データ移行そのものではなく、異種 DB のスキーマ変換支援が役割。移行シナリオの比較問題で出る。
- **どういう時に使うサービスか**: Oracle から PostgreSQL への移行準備、異種 DB 変換。

### AWS Secrets Manager
- **カテゴリ**: Security/Encryption
- **タグ**: `#セキュリティ` `#データ保護`
- **概要**: DB パスワードや API キーなどのシークレットを安全に保存・ローテーションするサービス。
- **SAA試験でのポイント**: 自動ローテーションが必要なら Secrets Manager が第一候補。Parameter Store との差は頻出。
- **どういう時に使うサービスか**: RDS 認証情報、外部 API キー、機密設定値の保管。

### AWS Security Hub
- **カテゴリ**: Security/Detection
- **タグ**: `#セキュリティ` `#統合管理`
- **概要**: 複数セキュリティサービスの検出結果を集約するダッシュボード。
- **SAA試験でのポイント**: 検知を自分でするサービスではなく、結果の統合と優先順位付けに強い。Organizations 配下での統合管理とも相性が良い。
- **どういう時に使うサービスか**: GuardDuty/Inspector/Macie の統合表示、セキュリティ運用センター。

### AWS Service Catalog
- **カテゴリ**: Security/Governance
- **タグ**: `#運用上の優秀性` `#標準化`
- **概要**: 承認済みの AWS リソースセットをカタログ化し、セルフサービス提供できるサービス。
- **SAA試験でのポイント**: 標準テンプレート配布やセルフサービス統制の文脈で出る。Service Catalog 自体が監査をするわけではない。
- **どういう時に使うサービスか**: 承認済み VPC の配布、標準開発環境のセルフサービス提供。

### AWS Service Quotas
- **カテゴリ**: Management/Operations
- **タグ**: `#信頼性` `#運用上の優秀性`
- **概要**: AWS サービスごとの利用上限を確認・管理するサービス。
- **SAA試験でのポイント**: 上限超過による障害予防の観点で Trusted Advisor と並んで出ることがある。性能改善サービスではなく、上限制御の可視化役。
- **どういう時に使うサービスか**: リソース上限監視、事前のクォータ引き上げ申請、運用リスク管理。

### AWS Shield
- **カテゴリ**: Security/Traffic
- **タグ**: `#セキュリティ` `#ネットワーク`
- **概要**: DDoS 攻撃から保護するサービス。
- **SAA試験でのポイント**: Standard は自動適用、Advanced は強化保護と DRT 支援が付く。WAF は L7 攻撃対策、Shield は DDoS 対策。
- **どういう時に使うサービスか**: 公開 Web サービスの DDoS 防御、ミッションクリティカル API の保護。

### AWS Site-to-Site VPN
- **カテゴリ**: Networking/Connectivity
- **タグ**: `#ネットワーク` `#ハイブリッド`
- **概要**: インターネット経由でオンプレミスと AWS を接続する VPN。
- **SAA試験でのポイント**: Direct Connect より安価だが、安定性と帯域は劣る。短期接続やコスト重視のハイブリッド構成でよく使う。
- **どういう時に使うサービスか**: 拠点接続、PoC、DR 用の予備回線。

### AWS Snow Family
- **カテゴリ**: Migration/Data
- **タグ**: `#移行` `#データ転送`
- **概要**: 物理デバイスで大容量データを AWS へ搬送するサービス群。
- **SAA試験でのポイント**: 数十 TB〜PB 級でネットワーク転送が現実的でない時の定番。Storage Gateway や DataSync との差分が重要。
- **どういう時に使うサービスか**: 初回データ移行、遠隔地の大量バックアップ搬送、帯域制約回避。

### Amazon SNS
- **カテゴリ**: AppIntegration/Messaging
- **タグ**: `#運用上の優秀性` `#疎結合` `#通知`
- **概要**: Pub/Sub 型のメッセージ配信サービス。複数の購読先へ同時通知できる。
- **SAA試験でのポイント**: SQS とのファンアウト構成が頻出。2026 時点ではメッセージデータ保護により PII/PHI の監査・マスキング・拒否も可能。
- **どういう時に使うサービスか**: メール/SMS 通知、イベントの同報配信、疎結合な通知ハブ。

### Amazon SQS
- **カテゴリ**: AppIntegration/Messaging
- **タグ**: `#運用上の優秀性` `#疎結合` `#耐障害性`
- **概要**: 非同期処理のためのメッセージキューサービス。
- **SAA試験でのポイント**: 標準は高スループット、FIFO は順序保証。CloudWatch 監視と Auto Scaling の連携もよく出る。
- **どういう時に使うサービスか**: バックエンドのバッファリング、注文処理の非同期化、ワーカー分散。

### AWS STS
- **カテゴリ**: Security/IAM
- **タグ**: `#セキュリティ` `#アクセス制御`
- **概要**: 一時的な認証情報を発行するサービス。
- **SAA試験でのポイント**: AssumeRole、フェデレーション、Cognito の ID プールなどの裏側で使われる。長期アクセスキー削減の本命。
- **どういう時に使うサービスか**: クロスアカウントアクセス、一時クレデンシャル発行、外部 ID 連携。

### AWS Step Functions
- **カテゴリ**: AppIntegration/Workflow
- **タグ**: `#サーバーレス` `#運用上の優秀性`
- **概要**: 複数サービスを状態遷移付きワークフローでつなぐサービス。
- **SAA試験でのポイント**: Lambda の連鎖を見やすく制御したい時の正解。分岐、再試行、人手承認などを実装しやすい。
- **どういう時に使うサービスか**: バッチ処理オーケストレーション、承認フロー、分散ワークフロー。

### AWS Storage Gateway
- **カテゴリ**: Storage/Hybrid
- **タグ**: `#ハイブリッド` `#Backup&Restore`
- **概要**: オンプレミスから AWS ストレージへ接続するゲートウェイサービス。
- **SAA試験でのポイント**: File / Volume / Tape Gateway の違いが基本。常設ハイブリッド連携でよく出る。
- **どういう時に使うサービスか**: バックアップ保管、既存アプリからの S3 連携、仮想テープ移行。

### AWS Systems Manager
- **カテゴリ**: Management/Operations
- **タグ**: `#運用上の優秀性` `#セキュリティ`
- **概要**: EC2 やオンプレサーバーの運用管理を一元化するサービス。
- **SAA試験でのポイント**: Patch Manager、Session Manager、Run Command、Parameter Store の使い分けが重要。踏み台なしの運用もよく問われる。
- **どういう時に使うサービスか**: パッチ適用、リモート実行、サーバ構成管理、運用自動化。

### AWS Transfer Family
- **カテゴリ**: Migration/Data
- **タグ**: `#データ転送` `#運用上の優秀性`
- **概要**: SFTP / FTPS / FTP プロトコルを使って S3 または EFS へファイルを転送できるフルマネージドサービス。
- **SAA試験でのポイント**: 既存の SFTP クライアントやワークフローをそのまま流用して AWS へ移行したい場合の正解。DataSync（エージェント常駐型・スケジュール同期）や Snow Family（物理輸送）との違いで出やすい。
- **どういう時に使うサービスか**: 取引先からの EDI ファイル受信、レガシー FTP サーバーの S3 移行、パートナー企業との SFTP ファイル連携。

### AWS Transit Gateway
- **カテゴリ**: Networking/Connectivity
- **タグ**: `#ネットワーク` `#マルチVPC` `#ハイブリッド`
- **概要**: 複数の VPC・オンプレミスネットワーク（VPN/Direct Connect）をハブ型で接続するリージョンレベルのネットワークルーター。
- **SAA試験でのポイント**: VPC Peering は 1 対 1 の接続で推移的なルーティング不可。VPC が 3 つ以上になると Transit Gateway のほうが管理しやすく、VPC 追加時のスケーラビリティも高い。マルチアカウント環境では Resource Access Manager（RAM）で TGW を共有する構成が頻出。
- **どういう時に使うサービスか**: マルチ VPC 環境の集中ルーティング、オンプレ接続の一元化（VPN/Direct Connect の集約）、クロスアカウントの VPC 接続共有。

### AWS Trusted Advisor
- **カテゴリ**: Management/Operations
- **タグ**: `#コスト最適化` `#セキュリティ` `#運用上の優秀性`
- **概要**: AWS 環境をベストプラクティス観点で点検するサービス。
- **SAA試験でのポイント**: 何かを自動で修復するより、改善提案を出す診断役。Compute Optimizer との役割差を押さえる。
- **どういう時に使うサービスか**: 不要リソース発見、公開ポート点検、サービス制限確認。

### Amazon VPC
- **カテゴリ**: Networking/Core
- **タグ**: `#ネットワーク` `#セキュリティ`
- **概要**: 論理的に分離された仮想ネットワークを作るサービス。
- **SAA試験でのポイント**: サブネット分割、ルートテーブル、NAT Gateway、VPC エンドポイント、SG/NACL の関係が SAA の中核。ここを外すと点が伸びにくい。
- **どういう時に使うサービスか**: AWS 上のネットワーク基盤、公開/非公開分離、セキュアなシステム配置。

### VPC Endpoint
- **カテゴリ**: Networking/Access
- **タグ**: `#セキュリティ` `#コスト最適化` `#ネットワーク`
- **概要**: AWS サービスへのアクセスをインターネット非経由で行うための接続機能。
- **SAA試験でのポイント**: S3/DynamoDB はゲートウェイ型、それ以外は多くがインターフェイス型。NAT Gateway コスト削減とセキュリティ強化の両方で出やすい。
- **どういう時に使うサービスか**: プライベートサブネットから S3 アクセス、PrivateLink 接続、閉域通信。

### VPC Endpoint Policy (VPC エンドポイントポリシー)
- **カテゴリ**: Security/ResourcePolicy
- **タグ**: `#セキュリティ` `#VPC`
- **概要**: VPC エンドポイントに付与できる IAM 形式のアクセスポリシー。
- **SAA試験でのポイント**: 通信の「経路（エンドポイント）」を通過できるリソースやアクションを制限する。特定の S3 バケットへのアクセスのみを許可する、特定のアカウントからのみアクセス可能にする、といった「出口対策」として非常に重要。
- **どういう時に使うサービスか**: VPC 内からの特定リソース限定アクセス、他アカウントへの情報持ち出し防止（情報漏えい対策）。

### VPC Flow Logs
- **カテゴリ**: Security/Detection
- **タグ**: `#セキュリティ` `#ネットワーク`
- **概要**: VPC 内の通信メタデータを記録する機能。
- **SAA試験でのポイント**: ネットワーク通信の追跡では有効だが、パケット本文までは見ない。GuardDuty や CloudTrail との役割差が大切。
- **どういう時に使うサービスか**: 通信経路の調査、アクセス分析、セキュリティ監査。

### AWS WAF
- **カテゴリ**: Security/Traffic
- **タグ**: `#セキュリティ` `#Web保護`
- **概要**: Web アプリケーションへの HTTP/S リクエストをルールベースで制御するサービス。
- **SAA試験でのポイント**: SQL インジェクション、XSS、レート制限、IP 制限が中心。ALB、CloudFront、API Gateway と連携する。
- **どういう時に使うサービスか**: Web 攻撃対策、Bot 制御、API の入口保護。

### AWS X-Ray
- **カテゴリ**: Management/Monitoring
- **タグ**: `#パフォーマンス効率` `#運用上の優秀性`
- **概要**: 分散アプリケーションのリクエストをトレースし、ボトルネックを可視化するサービス。
- **SAA試験でのポイント**: CloudWatch がメトリクス/ログ、X-Ray が分散トレース。マイクロサービスの遅延特定で選ばせる問題に強い。
- **どういう時に使うサービスか**: API 連鎖の遅延調査、マイクロサービス障害解析、依存関係可視化。

## その他の重要用語（五十音順）

### アシュームロール（AssumeRole）
- **カテゴリ**: Security/IAM
- **タグ**: `#セキュリティ` `#クロスアカウント`
- **概要**: STS を使って別の IAM ロールを一時的に引き受ける仕組み。
- **SAA試験でのポイント**: クロスアカウントアクセスの本命。長期アクセスキー共有より安全な設計として頻出する。
- **どういう時に使うサービスか**: 共有アカウント経由アクセス、監査ロール、運用者の一時昇格。

### ウォームスタンバイ
- **カテゴリ**: DR/Strategy
- **タグ**: `#Warmup Standby` `#信頼性`
- **概要**: DR 側で縮小版環境を常時稼働させる DR 戦略。
- **SAA試験でのポイント**: パイロットライトより高コストだが復旧が速い。「最小台数で常時稼働」がキーワード。
- **どういう時に使うサービスか**: 数分単位復旧が必要な業務システム、Web サービス DR。

### 加重ルーティング
- **カテゴリ**: Networking/DNS
- **タグ**: `#Route53` `#Warmup Standby`
- **概要**: Route 53 で各エンドポイントへ割り当てる流量比率を制御する方式。
- **SAA試験でのポイント**: Blue/Green、段階的移行、ウォームスタンバイで登場しやすい。単純なフェイルオーバーとは違う。
- **どういう時に使うサービスか**: 新旧環境の段階移行、A/B 配信、縮小版 DR 環境への一部流量送出。

### キャッシュアサイド（Cache Aside）
- **カテゴリ**: Database/InMemory
- **タグ**: `#ElastiCache` `#パフォーマンス効率`
- **概要**: キャッシュミス時のみ DB を読み、結果をキャッシュへ格納する方式。
- **SAA試験でのポイント**: ElastiCache 問題で最も出やすい戦略。キャッシュ汚染を抑えやすく、実務でも標準的。
- **どういう時に使うサービスか**: 商品詳細表示、ランキング参照、読取中心アプリ。

### クラスタープレイスメントグループ
- **カテゴリ**: Computing/General
- **タグ**: `#EC2` `#パフォーマンス効率`
- **概要**: EC2 インスタンスを近接配置し、低レイテンシ・高スループット通信を狙う配置方式。
- **SAA試験でのポイント**: HPC や分散計算で有効。耐障害性より性能重視である点を理解する。
- **どういう時に使うサービスか**: HPC、ノード間通信が多い分析基盤、MPI 系ワークロード。

### クロスリージョンリードレプリカ
- **カテゴリ**: Database/Relational
- **タグ**: `#RDS` `#Pilot Light`
- **概要**: 別リージョンに非同期複製する RDS の読み取り専用レプリカ。
- **SAA試験でのポイント**: DR 時には昇格して書き込み可能にする。Multi-AZ と違ってリージョン DR 用であり、通常は非同期。
- **どういう時に使うサービスか**: 低コスト DR、海外読取、災害時の昇格先。

### コスト配分タグ
- **カテゴリ**: Management/Cost
- **タグ**: `#コスト最適化` `#ガバナンス`
- **概要**: コストを部門・案件・環境ごとに分類して追跡するためのタグ。
- **SAA試験でのポイント**: Organizations や CUR と組み合わせて使う。請求可視化の基本設計として出やすい。
- **どういう時に使うサービスか**: 部門別原価管理、案件別課金、環境別コスト分析。

### コントロールプレーン
- **カテゴリ**: DR/Concepts
- **タグ**: `#運用設計` `#Multi Site`
- **概要**: リソースを作成・変更・削除する管理操作面。
- **SAA試験でのポイント**: DR の切替時はコントロールプレーン依存を減らし、Route 53 などのデータプレーン操作を優先する考え方が問われる。
- **どういう時に使うサービスか**: CloudFormation での構築、設定変更、自動化 API 呼び出し。

### サービスコントロールポリシー（SCP）
- **カテゴリ**: Security/Governance
- **タグ**: `#Organizations` `#セキュリティ`
- **概要**: Organizations 配下のアカウントに対し、使えるサービスや操作の上限を定めるポリシー。
- **SAA試験でのポイント**: 権限付与ではなく「上限を絞る」仕組み。IAM Allow があっても SCP が拒否すれば使えない。
- **どういう時に使うサービスか**: 本番アカウントで特定リージョン禁止、危険操作の抑止、標準ガードレール。

### 署名付き URL
- **カテゴリ**: Networking/CDN
- **タグ**: `#S3` `#CloudFront` `#セキュリティ`
- **概要**: 期限付きで限定公開コンテンツへアクセスさせる仕組み。
- **SAA試験でのポイント**: 非公開オブジェクトを一時公開したい時に定番。S3 直接か CloudFront 経由かで設計が変わる。
- **どういう時に使うサービスか**: 会員向けファイル配布、限定ダウンロード、動画視聴制御。

### セキュリティグループ（SG）
- **カテゴリ**: Networking/Core
- **タグ**: `#VPC` `#セキュリティ`
- **概要**: ENI / インスタンス単位に適用するステートフルファイアウォール。
- **SAA試験でのポイント**: 戻り通信は自動許可、拒否ルールは書けない、インスタンス単位という点が NACL との違い。
- **どういう時に使うサービスか**: EC2 や RDS の通信制御、アプリ層の許可設計。

### 地理的ルーティング
- **カテゴリ**: Networking/DNS
- **タグ**: `#Route53` `#グローバル`
- **概要**: ユーザーの地理的位置に応じて応答先を分ける Route 53 のルーティング方式。
- **SAA試験でのポイント**: 低レイテンシ最優先ではなく、地域ごとの法令対応やコンテンツ分離で使う。
- **どういう時に使うサービスか**: 国別配信制御、地域別サイト出し分け、法規制対応。

### データプレーン
- **カテゴリ**: DR/Concepts
- **タグ**: `#運用設計` `#Multi Site`
- **概要**: 既存リソースを使って実際にトラフィックを流す処理面。
- **SAA試験でのポイント**: Route 53 のフェイルオーバーはデータプレーンで動くため、障害時にも有効な切替手段として出やすい。
- **どういう時に使うサービスか**: DNS 切替、ロードバランサ経由通信、既存経路での運用継続。

### ネットワーク ACL（NACL）
- **カテゴリ**: Networking/Core
- **タグ**: `#VPC` `#セキュリティ`
- **概要**: サブネット単位に適用するステートレスフィルタ。
- **SAA試験でのポイント**: 明示的拒否ができ、戻り通信も個別許可が必要。特定 IP ブロックの定番。
- **どういう時に使うサービスか**: サブネット境界の粗い制御、悪性 IP の拒否、追加防御層。

### バケットポリシー (S3 Bucket Policy)
- **カテゴリ**: Security/ResourcePolicy
- **タグ**: `#S3` `#セキュリティ`
- **概要**: S3 バケット単位に付与するリソースベースポリシー。JSON 形式で「誰（Principal）」が「何（Action）」をすることを許可・拒否するかを定義する。
- **SAA試験でのポイント**: クロスアカウントアクセス、パブリックアクセスの拒否、HTTPS 接続の強制、特定の VPC からのみのアクセス制限などが頻出。バケット所有者とオブジェクト所有者が異なる場合の挙動（S3 Object Ownership）も 2026 年時点の重要事項。
- **どういう時に使うサービスか**: 外部組織へのデータ提供、セキュリティガードレールの適用、VPC エンドポイント経由のみのアクセス制限。

### バックアップ & リストア
- **カテゴリ**: DR/Strategy
- **タグ**: `#Backup&Restore` `#信頼性`
- **概要**: バックアップだけを保持し、災害時に環境を再作成して復元する DR 戦略。
- **SAA試験でのポイント**: 4 戦略の中で最も安価だが RTO/RPO は長い。開発環境や停止許容が大きい業務に向く。
- **どういう時に使うサービスか**: 低重要度システム、長時間停止を許容できる内部システム。

### パイロットライト
- **カテゴリ**: DR/Strategy
- **タグ**: `#Pilot Light` `#信頼性`
- **概要**: DB などコアだけを DR 側で常時稼働し、アプリ層は停止しておく DR 戦略。
- **SAA試験でのポイント**: 「コアのみ稼働」「最小コストでそこそこ短い復旧」がキーワード。ウォームスタンバイとの差を狙われやすい。
- **どういう時に使うサービスか**: 中規模業務システム、コストを抑えたい DR。

### フェイルオーバー
- **カテゴリ**: DR/Concepts
- **タグ**: `#信頼性` `#Multi Site`
- **概要**: 障害発生時に待機系へ切り替える仕組み。
- **SAA試験でのポイント**: 自動か手動か、同一リージョンかクロスリージョンか、DNS ベースか DB 昇格かを区別して問われる。
- **どういう時に使うサービスか**: DB 切替、リージョン障害時の切替、ロードバランサ障害回避。

### フェイルオーバールーティング
- **カテゴリ**: Networking/DNS
- **タグ**: `#Route53` `#Multi Site`
- **概要**: 正常系が落ちたら待機系へ送る Route 53 のルーティング方式。
- **SAA試験でのポイント**: ヘルスチェックとの組み合わせが前提。DR 問題では最頻出クラス。
- **どういう時に使うサービスか**: プライマリ/セカンダリ構成、リージョン DR、Web サービス切替。

### ヘルスチェック
- **カテゴリ**: DR/Concepts
- **タグ**: `#Route53` `#ELB` `#信頼性`
- **概要**: 対象リソースが正常かを確認する仕組み。
- **SAA試験でのポイント**: ELB はターゲット単位、Route 53 はエンドポイント/リージョン切替判断で使う。用途が違う。
- **どういう時に使うサービスか**: 自動フェイルオーバー、死活監視、異常系の切離し。

### マルチサイト
- **カテゴリ**: DR/Strategy
- **タグ**: `#Multi Site` `#信頼性`
- **概要**: 複数リージョンで本番同等環境を同時稼働させる DR 戦略。
- **SAA試験でのポイント**: RPO/RTO が最小だが最も高価。 Global Tables、Global Accelerator、Route 53、Aurora Global Database と組み合わせて問われやすい。
- **どういう時に使うサービスか**: ミッションクリティカルなグローバルサービス、停止許容がほぼない業務。

### ユーザープール
- **カテゴリ**: Security/Identity
- **タグ**: `#Cognito` `#セキュリティ`
- **概要**: Cognito でユーザー登録・サインイン・MFA などを担う認証機能。
- **SAA試験でのポイント**: ID プールとの違いを必ず押さえる。ユーザー管理と認証そのものを担うのがユーザープール。
- **どういう時に使うサービスか**: 会員登録、ログイン、パスワード管理、MFA。

### ライトスルー（Write Through）
- **カテゴリ**: Database/InMemory
- **タグ**: `#ElastiCache` `#パフォーマンス効率`
- **概要**: 更新時に DB とキャッシュを同時更新する方式。
- **SAA試験でのポイント**: 読取一貫性を保ちやすいが、書込負荷が高い場面では不利。 Cache Aside との比較で出る。
- **どういう時に使うサービスか**: 読取即時反映が重要な参照系キャッシュ。

### リードレプリカ
- **カテゴリ**: Database/Relational
- **タグ**: `#RDS` `#Aurora` `#パフォーマンス効率`
- **概要**: 読み取り専用の複製先 DB。
- **SAA試験でのポイント**: 読み取りスケールアウトと DR 補助が役割。Multi-AZ の待機系とは目的が違う。
- **どういう時に使うサービスか**: 読取負荷分散、分析系参照、DR 先の準備。

### レイテンシールーティング
- **カテゴリ**: Networking/DNS
- **タグ**: `#Route53` `#パフォーマンス効率`
- **概要**: ユーザーに最も低遅延なリージョンへ誘導する Route 53 の方式。
- **SAA試験でのポイント**: 地理的ルーティングとの違いが頻出。目的は「近い地域」ではなく「応答が速い先」。
- **どういう時に使うサービスか**: グローバル Web サービス、海外ユーザー向け低遅延配信。

### ロールベースアクセス制御用 SAML
- **カテゴリ**: Security/IAM
- **タグ**: `#IAMIdentityCenter` `#セキュリティ`
- **概要**: 異なる認証ドメイン間で認証・認可情報をやり取りする標準。
- **SAA試験でのポイント**: 社内 IdP と AWS の連携で出やすい。 LDAP そのものではなく、SSO 連携の標準として理解する。
- **どういう時に使うサービスか**: 企業 SSO、外部 IdP 連携、AWS へのフェデレーション。
