# AWS SAA 類似サービス比較チートシート（2026年版）

> 試験で混同しやすいサービス・機能を横断整理。「どれを選ぶか」の判断軸を中心に記載。

---

## 1. コンピューティング・サーバーレス系

### EC2 vs Lambda vs Fargate vs Elastic Beanstalk

| 観点 | EC2 | Lambda | Fargate | Elastic Beanstalk |
|------|-----|--------|---------|-------------------|
| **実行単位** | VM（インスタンス） | 関数（コード） | コンテナ | アプリケーション |
| **サーバー管理** | 自分で管理 | 不要 | 不要（ECS/EKS上） | 自動（裏はEC2） |
| **実行時間上限** | 無制限 | 最大15分 | 無制限 | 無制限 |
| **課金** | 起動時間 | 実行時間・回数 | CPU/メモリ使用量 | 裏のEC2リソース |
| **スケール** | 手動/ASG | 自動 | 自動 | 自動（ASG） |

**見分けのポイント**
- **15分超のバッチ処理** → Lambda不可 → EC2 or Fargate
- **コードのみで完結する短期処理** → Lambda
- **コンテナで動かしたいがサーバー管理不要** → Fargate
- **既存Webアプリをすぐデプロイしたい** → Elastic Beanstalk

## 2. コンテナ・オーケストレーション系

### ECS vs EKS

| 観点 | ECS | EKS |
|------|-----|-----|
| **オーケストレーター** | AWS独自 | Kubernetes |
| **学習コスト** | 低い | 高い（k8s知識必要） |
| **エコシステム** | AWS統合に強い | k8sツール群が使える |
| **起動方法** | EC2 or Fargate | EC2 or Fargate |
| **選ぶ場面** | AWSネイティブで十分 | 既存k8s環境の移行、マルチクラウド |

**見分けのポイント**
- 問題文に **「Kubernetes」** → EKS
- シンプルなコンテナ実行 → ECS
- サーバー管理なし → どちらでも **Fargate** と組み合わせ

## 3. ストレージ・データ移行系

### ブロック・ファイル・オブジェクトストレージ比較

| 観点 | S3 | EBS | EFS | FSx | インスタンスストア |
|------|----|----|-----|-----|----------------|
| **種別** | オブジェクト | ブロック | ファイル（NFS） | ファイル | ブロック（一時） |
| **接続** | HTTP/S | 単一EC2（通常） | 複数EC2同時 | 複数EC2同時 | 単一EC2（ローカル） |
| **永続性** | 永続 | 永続 | 永続 | 永続 | 停止/終了で消える |
| **特徴** | 無制限容量、耐久性 | 低レイテンシ | マネージドNFS | SMB/Lustre対応 | 最高IO・一時的 |

**見分けのポイント**
- **複数EC2で共有が必要** → EFS（Linux/NFS）or FSx（Windows/SMBならFSx for Windows、高性能HPC系はFSx for Lustre）
- **EC2の起動ディスク** → EBS
- **一時的な高速IO（再起動不要）** → インスタンスストア
- **静的ファイル・バックアップ・大容量保存** → S3

### 転送・ハイブリッド系

| サービス | 主な用途 | 特徴 |
|---------|---------|------|
| **DataSync** | オンプレ↔AWSファイル転送 | エージェントが必要、定期スケジュール転送 |
| **Storage Gateway** | オンプレとAWSのハイブリッド接続 | ファイル/ボリューム/テープの3種類 |
| **Snow Family** | 大量データの物理輸送 | ネットワーク帯域が足りない・閉鎖環境向け |
| **Transfer Family** | SFTP/FTP/FTPSでS3/EFSへ接続 | 既存FTPワークフローをAWSへ移行 |

**見分けのポイント**
- **「PB級・ネット帯域不足」** → Snow Family（Snowball/Snowball Edge/Snowmobile）
- **「既存のSFTPクライアントをそのまま使いたい」** → Transfer Family
- **「オンプレとS3/EFSを継続的に同期」** → DataSync
- **「オンプレのNASをAWSにバックアップしつつ手元でも使う」** → Storage Gateway

## 4. データベース・キャッシュ系

### RDB vs NoSQL

| 観点 | RDS（MySQL等） | Aurora | DynamoDB | DocumentDB |
|------|--------------|--------|----------|------------|
| **種別** | リレーショナル | リレーショナル | KV/ドキュメント | ドキュメント |
| **スケール** | 垂直スケール | 水平読取スケール | 水平・自動 | 垂直スケール |
| **Multi-AZ** | あり | 自動（ストレージ） | デフォルト | あり |
| **互換性** | MySQL/PostgreSQL等 | MySQL/PostgreSQL互換 | AWS独自 | MongoDB互換 |
| **特徴** | 標準的な運用 | 高可用・高性能 | サーバーレス・無制限スケール | 既存MongoDBの移行 |

**見分けのポイント**
- **「MongoDB互換」** → DocumentDB
- **「ミリ秒以下・無制限スケール・スキーマレス」** → DynamoDB
- **「高可用性が必要なRDB・RDSより上位」** → Aurora
- **「標準的なRDBをマネージドで」** → RDS

### グローバルDB・DR系

| サービス | 用途 | RPO/RTO感 |
|---------|------|----------|
| **Aurora Global Database** | Auroraのクロスリージョン展開 | RPO秒以下、RTO1分以内 |
| **DynamoDB Global Tables** | DynamoDBのマルチリージョン同期 | ほぼリアルタイム、Active-Active |
| **ElastiCache Global Datastore** | Redisのクロスリージョンレプリカ | 非同期レプリケーション |

**見分けのポイント**
- **「Aurora + クロスリージョンDR」** → Aurora Global Database
- **「DynamoDB + マルチリージョン・Active-Active」** → DynamoDB Global Tables
- **「Redis + グローバルキャッシュ」** → ElastiCache Global Datastore

### インメモリキャッシュ比較

| 観点 | ElastiCache Redis | ElastiCache Memcached | DAX |
|------|------------------|-----------------------|-----|
| **対象** | 汎用キャッシュ | 汎用キャッシュ | DynamoDB専用 |
| **永続化** | あり | なし | なし |
| **データ構造** | リスト/セット/ハッシュ等 | シンプルKV | KV |
| **クラスタリング** | あり（Cluster Mode） | あり（水平分散） | あり |
| **レプリケーション** | あり | なし | あり |

**見分けのポイント**
- **「DynamoDBの読取をマイクロ秒に」** → DAX（Dynamoの前段に置くだけ）
- **「セッション管理・Pub/Sub・永続化が必要」** → Redis
- **「単純なキャッシュで水平スケールだけできればいい」** → Memcached

## 5. ネットワーク・配信・ロードバランス系

### マルチVPC接続

| 観点 | VPC Peering | Transit Gateway | PrivateLink | VPCエンドポイント |
|------|------------|----------------|------------|----------------|
| **接続形態** | VPC間1対1 | ハブ型（多対多） | サービス提供側 | AWSサービスへ |
| **推移的ルーティング** | 不可 | 可能 | N/A | N/A |
| **スケール** | VPC数が増えると管理複雑 | 多数VPCに向く | - | - |
| **コスト** | 低い | やや高い（TGW料金） | - | - |

**見分けのポイント**
- **「VPCが3つ以上・推移的ルーティングが必要」** → Transit Gateway
- **「2つのVPCを単純につなぐ」** → VPC Peering
- **「他社のサービスをプライベートに提供・利用」** → PrivateLink
- **「S3/DynamoDBへインターネットを通らずアクセス」** → VPCエンドポイント（ゲートウェイ型）
- **「その他AWSサービスへプライベートアクセス」** → VPCエンドポイント（インターフェイス型＝PrivateLink）

### ロードバランサ比較

| 観点 | ALB | NLB | Gateway Load Balancer |
|------|-----|-----|----------------------|
| **レイヤー** | L7（HTTP/HTTPS） | L4（TCP/UDP） | L3/L4 |
| **主な用途** | Webアプリ・マイクロサービス | 高スループット・低レイテンシ | 仮想アプライアンス（FW等） |
| **特徴** | パスベース・ホストベースルーティング、Cognito認証 | 静的IP、最高性能 | セキュリティ製品のインライン配置 |

**見分けのポイント**
- **「URLパス/ホスト名で振り分け」** → ALB
- **「TCP・固定IP・超低レイテンシ」** → NLB
- **「IDS/IPS・ファイアウォールのトラフィック検査」** → Gateway Load Balancer

### グローバル・ハイブリッド接続

| 観点 | CloudFront | Global Accelerator | Site-to-Site VPN | Direct Connect |
|------|-----------|-------------------|-----------------|---------------|
| **目的** | コンテンツキャッシュ配信 | TCPレイテンシ最適化 | オンプレ暗号化接続 | 専用線接続 |
| **特徴** | エッジでキャッシュ、HTTP/S | Anycast IP、非HTTPも可 | インターネット経由、低コスト | 安定帯域、高コスト |
| **キャッシュ** | あり | なし | なし | なし |

**見分けのポイント**
- **「静的コンテンツの高速配信・キャッシュ」** → CloudFront
- **「HTTPに限らずグローバルに低レイテンシ」** → Global Accelerator
- **「オンプレとAWSをすぐにつなぐ（安価）」** → Site-to-Site VPN
- **「安定した専用帯域・コンプライアンス要件」** → Direct Connect

### VPCセキュリティ

| 観点 | セキュリティグループ（SG） | ネットワークACL（NACL） |
|------|------------------------|----------------------|
| **適用対象** | ENI（インスタンスレベル） | サブネット全体 |
| **ステートフル** | ステートフル（戻り通信自動許可） | ステートレス（戻り通信も明示必要） |
| **ルール種別** | 許可のみ | 許可・拒否両方 |
| **評価順** | 全ルールを評価 | 番号順（最初にマッチしたルールが適用） |

**見分けのポイント**
- **「特定IPを明示的に拒否したい」** → NACL（SGは許可のみ）
- **「インスタンス単位の細かい制御」** → SG
- **「二重防御（多層防御）」** → SGとNACL両方使う

## 6. セキュリティ・暗号化・監査系

### 脅威検知・設定管理・調査系

| サービス | 役割 | 主な入力ソース |
|---------|------|-------------|
| **GuardDuty** | 脅威検知（異常な通信・操作の検出） | VPC Flow Logs、CloudTrail、DNS |
| **Inspector** | 脆弱性スキャン（EC2/コンテナ/Lambda） | EC2エージェント、ECRイメージ |
| **Macie** | S3の機密データ（PII等）検出 | S3 |
| **Security Hub** | セキュリティ所見の一元集約・スコア化 | GuardDuty、Inspector等 |
| **Detective** | インシデントの根本原因調査 | GuardDuty所見、CloudTrail、VPC Flow Logs |

**見分けのポイント**
- **「不正アクセス・マルウェア検知」** → GuardDuty
- **「EC2の脆弱性パッチ確認」** → Inspector
- **「S3の個人情報漏洩リスク」** → Macie
- **「複数サービスのアラートをまとめて管理」** → Security Hub
- **「インシデント発生後の調査・可視化」** → Detective

### 暗号化・シークレット管理

| 観点 | KMS | CloudHSM | Secrets Manager | Parameter Store |
|------|-----|----------|----------------|----------------|
| **用途** | 鍵管理・暗号化 | ハードウェアセキュリティモジュール | シークレット管理・ローテーション | 設定値・シークレット保存 |
| **管理形態** | AWS共有マルチテナント | 専用ハードウェア（シングルテナント） | マネージド | マネージド |
| **コスト** | 低 | 高（専用HW） | 有料（ローテーション込み） | 無料階層あり |
| **自動ローテーション** | なし（鍵自体） | なし | あり | なし（手動実装） |

**見分けのポイント**
- **「コンプライアンス上の専用HSMが必要（FIPS 140-2 Level 3）」** → CloudHSM
- **「RDSパスワードの自動ローテーション」** → Secrets Manager
- **「アプリ設定値・環境変数の一元管理（費用を抑えたい）」** → Parameter Store
- **「S3/EBSのデータ暗号化」** → KMS

### 監査系

| サービス | 役割 | 記録内容 |
|---------|------|---------|
| **CloudTrail** | API操作ログ記録 | 誰が・いつ・何のAPIを呼んだか |
| **AWS Config** | リソース設定変更の追跡・準拠チェック | リソース設定の変化履歴・ルール評価 |
| **Artifact** | AWSのコンプライアンスレポート入手 | SOCレポート・PCI DSSレポート等 |

**見分けのポイント**
- **「誰がEC2を止めたか調べたい」** → CloudTrail
- **「セキュリティグループが変更されていないか継続監視」** → AWS Config
- **「AWS側のSOC報告書が欲しい」** → Artifact（自分でログを取るわけではない）

## 7. IAM・認証・AD連携系

### 認証・認可系

| 観点 | Cognito User Pool | Cognito Identity Pool | IAM Identity Center | STS AssumeRole |
|------|------------------|----------------------|--------------------|-----------------|
| **役割** | アプリ向けユーザー認証 | AWS一時認証の払い出し | SSO・マルチアカウント | 一時的なIAM権限委譲 |
| **ユーザー管理** | 自前（Cognitoが持つ） | 外部IdP連携 | IdP連携またはCognito User Pool | IAMロールへの委譲 |
| **主なユース** | Webアプリのサインイン | S3/DynamoDBへの直接アクセス | Organizations複数アカウントへのSSO | クロスアカウント・一時権限 |

**見分けのポイント**
- **「モバイルアプリのログイン画面」** → Cognito User Pool
- **「ログイン後にS3/DynamoDBに直接アクセスさせたい」** → Cognito Identity Pool（+STS）
- **「社員がAWS複数アカウントにSSOでログイン」** → IAM Identity Center
- **「Lambda/EC2から別アカウントのリソースを操作」** → STS AssumeRole

### AD連携系

| サービス | 用途 | Active Directory |
|---------|------|----------------|
| **AWS Managed Microsoft AD** | フルマネージドAD（AWS上に構築） | AWS上に独立したADを持つ・オンプレADと信頼関係 |
| **AD Connector** | オンプレADへのプロキシ | AWSにADデータを複製しない・オンプレADに転送 |
| **Simple AD** | 軽量なLDAP互換ディレクトリ | Sambaベース・小規模・ADの全機能不要 |

**見分けのポイント**
- **「オンプレのADをそのまま使いたい（複製不要）」** → AD Connector
- **「AWSでフルに使えるADが必要・オンプレADとの信頼関係も設定したい」** → AWS Managed Microsoft AD
- **「小規模でシンプルなLDAPがあればいい」** → Simple AD
- **「Windows WorkSpacesとAD連携」** → Managed Microsoft AD または AD Connector

## 8. メッセージング・イベント駆動・アプリ統合系

### SQS vs SNS vs EventBridge vs Kinesis Data Streams

| 観点 | SQS | SNS | EventBridge | Kinesis Data Streams |
|------|-----|-----|-------------|---------------------|
| **配信モデル** | キュー（Pull） | Pub/Sub（Push） | イベントバス（Push） | ストリーム（Pull） |
| **用途** | 疎結合・バッファリング | 1→多への通知 | AWS/SaaSイベントのルーティング | リアルタイムストリーミング |
| **順序保証** | 標準: なし / FIFO: あり | なし | なし | シャード内で保証 |
| **保持期間** | 最大14日 | なし（プッシュのみ） | なし | 最大365日 |
| **フィルタ** | なし | 属性フィルタ | ルールベース | なし |

**見分けのポイント**
- **「処理側が遅くても溜めておきたい（バッファ）」** → SQS
- **「1つのイベントを複数のサービスに同時通知」** → SNS
- **「AWSサービスの変更・SaaSイベントで自動トリガー」** → EventBridge
- **「ログ・クリックストリームをリアルタイム処理」** → Kinesis Data Streams
- **「順序が重要」** → SQS FIFO または Kinesis（シャード内）

## 9. コスト・請求管理系

### Cost Explorer vs Budgets vs Cost Anomaly Detection vs CUR

| サービス | 役割 | 主な用途 |

- **「急に費用が跳ね上がったら自動で検知してほしい」** → Cost Anomaly Detection
- **「Athenaで詳細なコスト分析・タグ別集計など生データが必要」** → CUR

## 10. 分析系

### Athena vs Redshift vs EMR vs OpenSearch

| 観点 | Athena | Redshift | EMR | OpenSearch |
|------|--------|----------|-----|-----------|
| **用途** | S3上のアドホッククエリ | DWH（大規模集計・BI） | Hadoopエコシステム処理 | ログ検索・全文検索 |
| **インフラ管理** | サーバーレス | クラスター（マネージド） | クラスター（マネージド） | クラスター（マネージド） |

- **「ログのキーワード検索・ダッシュボード（Kibana的UI）」** → OpenSearch

## 11. 可用性・DR設計（アーキテクチャ）

### DR4戦略比較

| 戦略 | RTO | RPO | コスト | 概要 |
|------|-----|-----|--------|------|
| **バックアップ&リストア** | 高（時間〜日単位） | 高（最後のバックアップ時点） | 最低 | S3等にバックアップ。障害時に再構築 |
| **パイロットライト** | 中（分〜時間） | 低〜中 | 低 | 最小限のコアサービスのみ起動状態で待機 |
| **ウォームスタンバイ** | 低（分単位） | 低 | 中 | 最小構成で常時稼働させ、障害時にスケールアップ |
| **マルチサイト（Active-Active）** | 最低（秒以下） | 最低（ほぼゼロ） | 最高 | 複数リージョンで完全稼働。即時フェイルオーバー |

**見分けのポイント**
- **「コスト最優先・RTO/RPOに余裕あり」** → バックアップ&リストア
- **「DBだけは動かしておきたい」** → パイロットライト
- **「すぐに切り替えたい・ある程度コスト許容」** → ウォームスタンバイ
- **「ダウンタイムゼロ・コスト度外視」** → マルチサイト

### バックアップ最適化・自動化

| サービス | 対象リソース | 自動化 | 管理スコープ | 主なユースケース |
|---------|------------|--------|------------|----------------|
| **EBS Snapshot** | EBSボリューム | 手動（または自前スクリプト） | 単一アカウント・単一リージョン | シンプルな単発バックアップ・AMI作成の基本操作 |
| **Data Lifecycle Manager (DLM)** | EC2インスタンス・EBSボリューム | 自動（スケジュール設定） | 単一アカウント・EC2/EBS特化 | EC2/EBSのスナップショットライフサイクルを自動管理 |
| **AWS Backup** | EC2・EBS・RDS・DynamoDB・EFS・FSx・S3等 | 自動（バックアッププラン） | クロスアカウント・複数リソース一元管理 | 複数サービス・複数アカウントのバックアップを統合管理 |

**見分けのポイント**
- **「手動で単純にEBSを記録しておきたい」** → EBS Snapshot
- **「EC2/EBSのバックアップを自動スケジュール化したい（リソース対象が限定的）」** → Data Lifecycle Manager
- **「RDS・DynamoDBなど複数サービスを一括管理、または複数アカウント横断でバックアップポリシーを統制したい」** → AWS Backup

### Route 53 ルーティングポリシー

| ポリシー | 用途 | 特徴 |
|---------|------|------|
| **フェイルオーバー** | DR・HA | ヘルスチェック失敗時にセカンダリへ切替 |
| **加重（Weighted）** | A/Bテスト・段階デプロイ | 重みの比率でトラフィックを分散 |
| **レイテンシ（Latency）** | レイテンシ最小化 | クライアントに最も低レイテンシなリージョンへ |
| **地理的（Geolocation）** | 地域別コンテンツ・規制対応 | ユーザーの国・地域に基づいてルーティング |
| **地理的近接性（Geoproximity）** | 地理的距離＋バイアス調整 | Route 53 Traffic Flow必須・バイアスで範囲調整可能 |
| **複数値回答（Multivalue Answer）** | 簡易LB・可用性向上 | 複数のIPを返す・ヘルスチェックと組み合わせ |

**見分けのポイント**
- **「障害時に自動切替」** → フェイルオーバー
- **「新バージョンへ徐々に移行したい」** → 加重
- **「日本ユーザーは日本リージョン、米ユーザーは米リージョン（レイテンシ重視）」** → レイテンシ
- **「EUユーザーはEUのエンドポイントに固定（規制・法令対応）」** → 地理的
- **「地域ごとの流量をバイアスで細かく調整したい」** → 地理的近接性
- **「ELBの代わりに複数IPを返してヘルスチェックも行いたい（シンプル）」** → 複数値回答
