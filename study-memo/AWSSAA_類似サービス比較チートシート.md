# AWS SAA 類似サービス比較チートシート（2026年版・監査済み）

> 試験で混同しやすいサービス・機能を、「どれを選ぶか」「なぜ他が違うか」「問題文のどの語で見抜くか」で横断整理。
> 監査で修正した代表例: `NLB = TCP/UDP/TLS 系 + 固定 IP`、`KMS = 自動ローテーション可`、`DataSync = 常にエージェント必須ではない`、`SNS = FIFO あり`、`IAM Identity Center = Cognito User Pool を ID ソースにしない`、`DocumentDB = 読み取り水平拡張あり` など。

## 1. コンピューティング・サーバーレス系

### EC2 vs Lambda vs Fargate vs Elastic Beanstalk

| 観点 | EC2 | Lambda | Fargate | Elastic Beanstalk |
|------|-----|--------|---------|-------------------|
| **実行単位** | VM（インスタンス） | 関数（コード） | コンテナ | アプリケーション環境 |
| **サーバー管理** | 自分で管理 | 不要 | 不要（ECS/EKS 上） | 大半を自動化（裏は EC2/ALB/ASG） |
| **実行時間上限** | 無制限 | 最大 15 分 | 無制限 | 無制限 |
| **課金イメージ** | インスタンス稼働 | リクエスト数 + 実行時間 | vCPU/メモリ/実行時間 | 裏の EC2/EBS/ALB 等 |
| **向く処理** | 常駐、長時間、特殊要件 | 短時間イベント処理 | コンテナ化済みアプリ | 既存 Web アプリを手早く PaaS 風に運用 |

#### サービス別の見分け方と出題例

- **EC2**: OS レベルの設定変更、独自エージェント導入、SSH 保守、長時間バッチ、特殊ドライバやライセンスが必要なら本命。出題例: 「サードパーティ製ソフトをインストールし、毎晩 3 時間の処理を実行。カーネル設定も変更したい」→ EC2。
- **Lambda**: 「イベント駆動」「短時間」「サーバー管理不要」「アクセスがないときは課金を抑えたい」が見えたら最有力。出題例: 「S3 に画像が置かれたら自動でサムネイルを作成。通常はほぼ実行されない」→ Lambda。
- **Fargate**: コンテナ要件はあるがノード管理はしたくない、というときの答え。出題例: 「Docker 化された API を ECS で動かしたいが、EC2 キャパシティや AMI 更新を考えたくない」→ Fargate。
- **Elastic Beanstalk**: アプリをすぐにデプロイしたいが、コード中心で環境構築を簡略化したいときに刺さる。出題例: 「Java Web アプリを最小運用負荷でデプロイし、Auto Scaling と負荷分散も自動設定したい」→ Elastic Beanstalk。

## 2. コンテナ・オーケストレーション系

### ECS vs EKS

| 観点 | ECS | EKS |
|------|-----|-----|
| **オーケストレーター** | AWS 独自 | Kubernetes |
| **学習コスト** | 低い | 高い |
| **AWS 統合** | 非常に強い | 強い |
| **可搬性** | AWS 寄り | Kubernetes 標準に乗りやすい |
| **起動方法** | EC2 or Fargate | EC2 or Fargate |

#### サービス別の見分け方と出題例

- **ECS**: 問題文に Kubernetes が出ず、AWS ネイティブでシンプルにコンテナ実行したいなら ECS を優先する。出題例: 「社内チームは k8s 経験が薄く、AWS のマネージドサービスと素直に統合したい」→ ECS。
- **EKS**: 「既存 Kubernetes マニフェスト」「Helm」「複数クラウドで統一」「k8s エコシステム活用」が見えたら EKS。出題例: 「オンプレ Kubernetes のワークロードをできるだけ同じ運用モデルで AWS に移したい」→ EKS。

## 3. ストレージ・データ移行系

### ブロック・ファイル・オブジェクトストレージ比較

| 観点 | S3 | EBS | EFS | FSx | インスタンスストア |
|------|----|-----|-----|-----|-------------------|
| **種別** | オブジェクト | ブロック | ファイル（NFS） | マネージドファイル | ローカルブロック |
| **共有性** | HTTP/S 経由 | 通常は単一 EC2 にアタッチ | 複数 EC2 で共有 | ワークロード別に共有可能 | 単一ホスト |
| **永続性** | 永続 | 永続 | 永続 | 永続 | インスタンス停止/終了で消失 |
| **代表用途** | 静的配信、バックアップ、データレイク | 起動ディスク、DB ディスク | 共有ファイル | Windows/NetApp/Lustre など特化要件 | キャッシュ、スクラッチ領域 |

#### サービス別の見分け方と出題例

- **S3**: 「無制限」「オブジェクト」「静的ファイル」「バックアップ」「ログ保管」「イベント通知」が出たら S3。出題例: 「Web サイトの画像を高耐久かつ低コストで保管し、必要ならライフサイクルでアーカイブしたい」→ S3。
- **EBS**: 単一 EC2 に低レイテンシな永続ブロックストレージを付ける場面で選ぶ。出題例: 「EC2 上のリレーショナル DB 用に、再起動や置換後も残るディスクが必要」→ EBS。
- **EFS**: Linux 系インスタンスから同時マウントできる共有ファイルストレージが必要なら EFS。出題例: 「複数 AZ の Web サーバー群でユーザーアップロードを共用したい」→ EFS。
- **FSx**: プロトコルやワークロードが明確なときに選ぶサービス群で、Windows SMB、Lustre、NetApp ONTAP、OpenZFS などが対象。出題例: 「Windows サーバー群から SMB 共有が必要」→ FSx for Windows File Server。
- **インスタンスストア**: 最高速だが揮発性でよいデータ向け。出題例: 「再生成できる中間ファイルやキャッシュを、できるだけ高速なローカルディスクに置きたい」→ インスタンスストア。

### S3 ストレージクラス比較

| ストレージクラス | 向く用途 | 取り出し特性 |
|------------------|----------|--------------|
| **Standard** | 頻繁にアクセスする本番データ | 即時 |
| **Intelligent-Tiering** | アクセス頻度が読めないデータ | 即時（アーカイブ層を除く） |
| **Standard-IA** | 低頻度アクセスだが必要時は即時 | 即時 |
| **One Zone-IA** | 再作成可能で AZ 単位冗長を落としてよい低頻度データ | 即時 |
| **Glacier Instant Retrieval** | まれに読む長期保管データ | ミリ秒 |
| **Glacier Flexible Retrieval** | 取り出しに分〜時間を許容できるアーカイブ | 即時ではない |
| **Glacier Deep Archive** | ほぼ読まない超長期保管 | 時間単位 |

#### ストレージクラス別の見分け方と出題例

- **S3 Standard**: 頻繁アクセスの現役データ、静的サイト配信元、アプリ本番データならまずこれ。出題例: 「モバイルアプリが日常的に参照する画像を保存」→ Standard。
- **S3 Intelligent-Tiering**: アクセスパターンが読めず、自動最適化したいときに強い。出題例: 「どのファイルが来月も読まれるか予測できない文書保管庫」→ Intelligent-Tiering。
- **S3 Standard-IA**: 低頻度アクセスだが、必要時は即取り出したいバックアップや DR コピー向け。出題例: 「月 1 回程度参照する監査ログ」→ Standard-IA。
- **S3 One Zone-IA**: コスト重視で、AZ 障害時に失っても再作成できるデータ向け。出題例: 「他リージョンに原本がある一時コピー」→ One Zone-IA。
- **S3 Glacier Instant Retrieval**: アーカイブ寄りだがミリ秒アクセスは欲しい、という設問で使う。出題例: 「医療画像を長期保管するが、急な参照にも即応したい」→ Glacier Instant Retrieval。
- **S3 Glacier Flexible Retrieval**: 取り出しが数分〜時間でよければコストをさらに下げられる。出題例: 「年次監査時だけ参照するアーカイブ」→ Glacier Flexible Retrieval。
- **S3 Glacier Deep Archive**: 最安クラスで、取り出しを急がない法定保存や長期バックアップ向け。出題例: 「7 年保存が必要だが、ほぼ閲覧しない」→ Glacier Deep Archive。

### 転送・ハイブリッド系

| サービス | 主な用途 | 特徴 |
|---------|---------|------|
| **DataSync** | オンプレ/他クラウド/AWS 間のデータ転送 | 高速転送、スケジュール、検証、AWS 間転送ではエージェント不要なケースあり |
| **Storage Gateway** | オンプレと AWS ストレージの橋渡し | File/Volume/Tape 系のハイブリッド接続 |
| **Snow Family** | 大量データの物理輸送 | 帯域不足、閉域、遠隔地に強い |
| **Transfer Family** | SFTP/FTP/FTPS で S3/EFS に接続 | 既存ファイル転送ワークフロー移行向け |

#### サービス別の見分け方と出題例

- **DataSync**: 継続的なデータ移行・同期・増分コピー・整合性検証が欲しいときに選ぶ。出題例: 「オンプレ NAS のデータを毎晩 EFS に同期したい」→ DataSync。
- **Storage Gateway**: オンプレ側から既存プロトコルで使いながら、裏を AWS に逃がしたいときの答え。出題例: 「既存バックアップソフトはテープ前提だが、実体は AWS に置きたい」→ Tape Gateway。
- **Snow Family**: 「数百 TB〜PB」「回線が細い」「船舶・工場・遠隔地」「オフライン搬送」が出たら Snow を疑う。出題例: 「ペタバイト級映像データを短期間で AWS に移したい」→ Snowball Edge や Snowmobile。
- **Transfer Family**: 既存の SFTP/FTP クライアントや取引先連携を変えずに AWS に持ち込みたいときに刺さる。出題例: 「取引先は SFTP しか話せないが、保存先は S3 にしたい」→ Transfer Family。

## 4. データベース・キャッシュ系

### RDB vs NoSQL

| 観点 | RDS（MySQL/PostgreSQL 等） | Aurora | DynamoDB | DocumentDB |
|------|---------------------------|--------|----------|------------|
| **種別** | リレーショナル DB | クラウドネイティブ RDB | キー値/ドキュメント | ドキュメント DB |
| **スケール** | 主に垂直、読取レプリカあり | 高性能、読取スケール、ストレージ自動拡張 | 水平・自動 | 読取レプリカで読取拡張 |
| **可用性** | Multi-AZ 構成可 | ストレージは 3 AZ に 6 コピー | マルチ AZ 前提サービス | 6 コピーのクラスタボリューム |
| **互換性** | 各エンジンそのもの | MySQL/PostgreSQL 互換 | AWS 独自 API | MongoDB 互換 |

#### サービス別の見分け方と出題例

- **RDS**: 一般的な RDB をマネージド化したいが、必ずしも Aurora 級の性能や設計変更は要らない場面で選ぶ。出題例: 「既存の MySQL アプリを最小変更でマネージド移行したい」→ RDS for MySQL。
- **Aurora**: 「高可用」「高性能」「RDS より読取性能」「グローバル DB」「高速フェイルオーバー」が見えたら最有力。出題例: 「ミッションクリティカルな PostgreSQL 互換 DB を高可用で運用したい」→ Aurora PostgreSQL。
- **DynamoDB**: キー設計中心、ミリ秒未満〜数ミリ秒、サーバーレス、急なスパイク、巨大スケールなら DynamoDB。出題例: 「ゲームのセッション状態を超高スループットで保存し、運用負荷を極小化したい」→ DynamoDB。
- **DocumentDB**: 「MongoDB 互換」「JSON ドキュメント」「既存 Mongo ドライバ資産」が問題文にあると候補になる。出題例: 「MongoDB ベースのアプリをマネージド化したい」→ DocumentDB。

### グローバル DB・DR 系

| サービス | 用途 | 特徴 |
|---------|------|------|
| **Aurora Global Database** | Aurora のクロスリージョン DR/読取分散 | 低 RPO、短い RTO |
| **DynamoDB Global Tables** | DynamoDB のマルチリージョン Active-Active | ほぼリアルタイム複製 |
| **ElastiCache Global Datastore** | Redis/Valkey のグローバル読取 | 非同期レプリケーション |

#### サービス別の見分け方と出題例

- **Aurora Global Database**: RDB でクロスリージョン DR を短い RTO/RPO で求めるならこれ。出題例: 「北米リージョン障害時、1 分未満で Aurora を昇格したい」→ Aurora Global Database。
- **DynamoDB Global Tables**: 各リージョンからローカル書き込みしたい Active-Active 要件なら本命。出題例: 「世界各地のユーザーが最寄りリージョンに書き込み、同一テーブルを共有したい」→ DynamoDB Global Tables。
- **ElastiCache Global Datastore**: グローバルキャッシュや DR 用リードレプリカの文脈で出る。出題例: 「海外リージョンにも低遅延で参照させたい Redis キャッシュ」→ ElastiCache Global Datastore。

### インメモリキャッシュ比較

| 観点 | ElastiCache for Redis/Valkey | ElastiCache for Memcached | DAX |
|------|------------------------------|---------------------------|-----|
| **対象** | 汎用キャッシュ、データ構造活用 | 単純キャッシュ | DynamoDB 専用キャッシュ |
| **永続化** | あり | なし | なし |
| **レプリケーション** | あり | なし | あり |
| **得意分野** | セッション、Pub/Sub、ランキング | 単純で軽い分散キャッシュ | DynamoDB 読取高速化 |

#### サービス別の見分け方と出題例

- **Redis/Valkey**: セッション共有、Pub/Sub、Sorted Set、永続化、レプリケーションが必要ならこちら。出題例: 「EC サイトのログインセッションとランキングを同じキャッシュ基盤で持ちたい」→ ElastiCache for Redis/Valkey。
- **Memcached**: 単純なキー値キャッシュを水平分散できれば十分で、永続化も複雑な機能も不要なとき向き。出題例: 「DB クエリ結果の使い捨てキャッシュだけを低コストで持ちたい」→ Memcached。
- **DAX**: DynamoDB の読み取り遅延をマイクロ秒級に下げたい、という設問で見抜く。出題例: 「既存 DynamoDB のコード変更を最小にして読取性能だけ上げたい」→ DAX。

## 5. ネットワーク・配信・ロードバランス系

### マルチ VPC 接続

| 観点 | VPC Peering | Transit Gateway | PrivateLink | VPC エンドポイント |
|------|-------------|----------------|-------------|-------------------|
| **接続形態** | VPC 間 1 対 1 | ハブ型 | サービス単位の私設接続 | AWS サービス/自社サービスへの私設接続 |
| **推移的ルーティング** | 不可 | 可能 | N/A | N/A |
| **向く規模** | 少数 VPC | 多数 VPC/ハイブリッド | サービス提供/利用 | 個別サービスアクセス |
| **代表用途** | シンプル接続 | 多拠点集約 | SaaS/共有サービス公開 | S3/DynamoDB/各種 AWS API へ私設アクセス |

#### サービス別の見分け方と出題例

- **VPC Peering**: VPC が少数で、単純に 2 つをつなぐだけなら最も素直。出題例: 「開発 VPC と共有サービス VPC のみを低コストで接続したい」→ VPC Peering。
- **Transit Gateway**: VPC 数が増え、接続関係がメッシュ化して管理破綻しそうなら TGW。出題例: 「数十の VPC とオンプレ接続を 1 か所で集約したい」→ Transit Gateway。
- **PrivateLink**: 相手 VPC 全体ではなく、特定サービスだけをプライベートに公開・利用したいときに選ぶ。出題例: 「SaaS 提供者として顧客 VPC から自社サービスへ私設接続させたい」→ PrivateLink。
- **VPC エンドポイント**: AWS サービスへインターネットを経由せずにアクセスしたいときの総称。出題例: 「プライベートサブネットの EC2 から S3 に NAT なしでアクセスしたい」→ Gateway Endpoint。

### ロードバランサ比較

| 観点 | ALB | NLB | Gateway Load Balancer |
|------|-----|-----|----------------------|
| **レイヤー** | L7（HTTP/HTTPS/gRPC） | L4（TCP/UDP/TLS） | L3/L4 |
| **得意分野** | URL/ホストベース振り分け | 超低レイテンシ、高スループット、固定 IP | 仮想アプライアンス透過配置 |
| **特徴** | パス/ホスト条件、認証連携 | 静的 IP、ソース IP 保持、TLS/UDP 対応 | セキュリティ製品をインライン挿入 |

#### サービス別の見分け方と出題例

- **ALB**: Web アプリで「パスベース」「ホストベース」「HTTP ヘッダー」「WebSocket/gRPC」「Lambda ターゲット」が出たら ALB。出題例: 「`/api` は API 群へ、`/images` は別ターゲットへ振り分けたい」→ ALB。
- **NLB**: 固定 IP、超低レイテンシ、TCP/UDP/TLS、急激なスパイク耐性が欲しいときに選ぶ。出題例: 「UDP ベースのゲームトラフィックを固定 IP で受けたい」→ NLB。
- **Gateway Load Balancer**: ファイアウォール、IDS/IPS、DPI 製品などを通過させたい文脈ならこれ。出題例: 「複数 VPC の通信をサードパーティ製 FW で一元検査したい」→ Gateway Load Balancer。

### グローバル・ハイブリッド接続

| 観点 | CloudFront | Global Accelerator | Site-to-Site VPN | Direct Connect |
|------|-----------|-------------------|-----------------|---------------|
| **目的** | HTTP/S コンテンツ配信 | TCP/UDP のグローバル最適化 | 暗号化トンネル | 専用線 |
| **特徴** | エッジキャッシュ、署名 URL/WAF と相性良 | Anycast IP、最寄りエッジ入口 | すぐ始められる、安価 | 安定帯域、予測しやすい性能 |
| **キャッシュ** | あり | なし | なし | なし |

#### サービス別の見分け方と出題例

- **CloudFront**: キャッシュ、静的/動的コンテンツ配信、Signed URL/Cookie、エッジ最適化が必要なら CloudFront。出題例: 「世界中のユーザーへ画像と動画を低遅延で配信したい」→ CloudFront。
- **Global Accelerator**: キャッシュではなく、グローバルな入口 IP と TCP/UDP 経路最適化が欲しいときに選ぶ。出題例: 「複数リージョンの ALB/NLB へ単一 Anycast IP で振り分けたい」→ Global Accelerator。
- **Site-to-Site VPN**: まず早く安く暗号化接続したいときの標準解。出題例: 「今週中にオンプレと AWS を接続したいが専用線は間に合わない」→ Site-to-Site VPN。
- **Direct Connect**: 安定した帯域、低揺らぎ、長期利用、コンプライアンス要件が重いなら DX。出題例: 「常時大容量データをオンプレから AWS へ流し、インターネットを避けたい」→ Direct Connect。

### インターネット接続経路の見分け方

| 観点 | Internet Gateway | NAT Gateway | Egress-only Internet Gateway |
|------|------------------|-------------|------------------------------|
| **対象** | パブリックサブネット | プライベートサブネットの外向き IPv4 | プライベートサブネットの外向き IPv6 |
| **受信** | 可能 | 不可 | 不可 |
| **主な用途** | 公開 Web サーバー | パッチ取得、外部 API 呼び出し | IPv6 の外向き専用通信 |

#### サービス別の見分け方と出題例

- **Internet Gateway**: パブリック IP を持つリソースをインターネット公開するときに使う。出題例: 「公開 ALB を VPC にぶら下げて外部ユーザーからアクセスさせたい」→ Internet Gateway。
- **NAT Gateway**: プライベートサブネットの EC2/Lambda から外へは出したいが、外からは入れたくないときの答え。出題例: 「アプリサーバーは非公開のまま OS パッチを取得したい」→ NAT Gateway。
- **Egress-only Internet Gateway**: IPv6 版の外向き専用出口として出る。出題例: 「IPv6 のみのプライベートサブネットから外部へ更新通信を出したい」→ Egress-only Internet Gateway。

### VPC セキュリティ

| 観点 | セキュリティグループ（SG） | ネットワーク ACL（NACL） |
|------|---------------------------|--------------------------|
| **適用対象** | ENI/インスタンス | サブネット |
| **状態管理** | ステートフル | ステートレス |
| **ルール種別** | 許可のみ | 許可・拒否 |
| **評価** | すべての許可ルールの総和 | ルール番号順に最初に一致したもの |

#### サービス別の見分け方と出題例

- **セキュリティグループ**: 基本はまず SG。インスタンス単位で許可ベース制御をしたい設問で選ぶ。出題例: 「Web サーバーだけ 443 を許可し、DB にはアプリサーバーからの 3306 のみ許可」→ SG。
- **NACL**: 明示的 deny が必要、サブネット単位の粗い境界制御が必要なら NACL。出題例: 「特定の送信元 IP レンジを明示的に拒否したい」→ NACL。

## 6. セキュリティ・暗号化・監査系

### 脅威検知・設定管理・調査系

| サービス | 役割 | 主な対象 |
|---------|------|----------|
| **GuardDuty** | 脅威検知 | アカウント、VPC、EKS、S3、RDS など |
| **Inspector** | 脆弱性管理/露出診断 | EC2、ECR、Lambda |
| **Macie** | S3 機密データ検出 | S3 |
| **Security Hub** | 所見の集約/標準準拠 | 各種セキュリティサービス |
| **Detective** | 事後調査・関係分析 | GuardDuty 所見など |

#### サービス別の見分け方と出題例

- **GuardDuty**: 「脅威検知」「異常 API 呼び出し」「怪しい通信」「マルウェア兆候」なら GuardDuty。出題例: 「盗まれた認証情報による不審な API 利用を検出したい」→ GuardDuty。
- **Inspector**: 「脆弱性」「CVE」「未修正パッケージ」「公開ポート露出評価」が見えたら Inspector。出題例: 「ECR イメージと EC2 を継続的にスキャンしたい」→ Inspector。
- **Macie**: S3 内の PII、機密文書、個人情報を見つけたいときに選ぶ。出題例: 「S3 バケットに保存された顧客 CSV の個人情報漏えいリスクを把握したい」→ Macie。
- **Security Hub**: GuardDuty や Inspector の所見を 1 か所に集めて優先度付けしたいなら Security Hub。出題例: 「複数アカウントのセキュリティ所見を統合ダッシュボードで管理したい」→ Security Hub。
- **Detective**: インシデント後の因果関係調査、アクティビティの掘り下げに使う。出題例: 「GuardDuty の所見が出た後、どのロールが何にアクセスしたかを時系列で調べたい」→ Detective。

### 暗号化・シークレット管理

| 観点 | KMS | CloudHSM | Secrets Manager | Parameter Store |
|------|-----|----------|----------------|----------------|
| **用途** | 鍵管理と暗号化 API | 専用 HSM | シークレット保管とローテーション | 設定値/シークレット保管 |
| **運用形態** | フルマネージド | 専用 HSM クラスタ運用 | フルマネージド | フルマネージド |
| **自動ローテーション** | 対象キーで可 | なし | 可 | 標準機能では不可 |
| **向く場面** | S3/EBS/RDS 暗号化全般 | 専有 HSM 必須 | DB パスワードや API キー | 安価な設定集中管理 |

#### サービス別の見分け方と出題例

- **KMS**: AWS サービス統合で暗号化鍵を管理したいならまず KMS。出題例: 「S3 と EBS を顧客管理キーで暗号化し、アクセス制御と監査も行いたい」→ KMS。
- **CloudHSM**: 専有ハードウェア HSM、より強いコンプライアンス、独自暗号要件が必要ならこちら。出題例: 「共有マルチテナント HSM では要件を満たせず、専用 HSM が必要」→ CloudHSM。
- **Secrets Manager**: シークレットの自動ローテーションが問題文に出たら最優先。出題例: 「RDS の認証情報を 30 日ごとに自動更新したい」→ Secrets Manager。
- **Parameter Store**: 設定値や機密値を階層管理したいが、ローテーション要件は薄くコストを抑えたいとき向き。出題例: 「アプリ設定と SecureString を一元管理したい」→ Parameter Store。

### 監視・監査系

| サービス | 主目的 | 典型質問 |
|---------|--------|----------|
| **CloudWatch** | メトリクス/ログ/アラーム監視 | 今の状態はどうか |
| **CloudTrail** | API 操作履歴 | 誰が何をしたか |
| **AWS Config** | 設定変更履歴と準拠評価 | あるべき設定から逸脱していないか |
| **Artifact** | AWS の監査レポート入手 | AWS 側の証跡資料を入手したい |

#### サービス別の見分け方と出題例

- **CloudWatch**: CPU 使用率、エラーレート、アラーム、ログ監視、ダッシュボードなら CloudWatch。出題例: 「Lambda のエラー回数が閾値を超えたら SNS 通知したい」→ CloudWatch Alarm。
- **CloudTrail**: 監査ログで「誰が EC2 を停止したか」「どの API が呼ばれたか」を知りたいときに使う。出題例: 「本番 SG を変更した IAM ユーザーを特定したい」→ CloudTrail。
- **AWS Config**: 現在/過去のリソース設定、ルール準拠、ドリフト監視がテーマなら Config。出題例: 「S3 バケットが公開設定になったら即検知したい」→ AWS Config。
- **Artifact**: AWS 自身の SOC/PCI/DSS などのレポートが欲しいときの窓口。出題例: 「監査部門へ AWS のコンプライアンス報告書を提出したい」→ Artifact。

| 観点 | Cognito User Pool | Cognito Identity Pool | IAM Identity Center | STS AssumeRole |

- **STS AssumeRole**: AWS 内の権限委譲、特にクロスアカウントや一時権限取得ならこれ。出題例: 「監査アカウントの Lambda から本番アカウントの S3 を読ませたい」→ STS AssumeRole。

## 8. メッセージング・イベント駆動・アプリ統合系

### SQS vs SNS vs EventBridge vs Kinesis Data Streams

| 観点 | SQS | SNS | EventBridge | Kinesis Data Streams |
|------|-----|-----|-------------|---------------------|
| **モデル** | キュー | Pub/Sub | イベントバス | ストリーム |
| **配信** | Pull | Push | Push | Pull |
| **順序** | FIFO で可 | FIFO Topic で可 | なし | シャード内で保証 |
| **主用途** | バッファリング/疎結合 | 一斉通知 | イベントルーティング | リアルタイム分析 |
| **保持** | 最大 14 日 | 配信中心 | 標準では保持前提でない（必要なら Archive） | 最大 365 日 |

#### サービス別の見分け方と出題例

- **SQS**: コンシューマの処理速度と切り離したい、再試行したい、ワーカーに順番に読ませたいなら SQS。出題例: 「注文処理が一時的に詰まってもリクエストを失わず後続処理したい」→ SQS。
- **SNS**: 1 つのイベントを複数宛先へ即時配信したいときに使う。出題例: 「注文確定をメール通知、Lambda 起動、SQS 投入へ同時に配りたい」→ SNS。
- **EventBridge**: AWS サービスイベントや SaaS イベントをルールベースで振り分けたいときに強い。出題例: 「EC2 状態変更イベントを受けて Step Functions を起動したい」→ EventBridge。
- **Kinesis Data Streams**: 高頻度データを順序付きでリアルタイム処理したいなら KDS。出題例: 「クリックストリームを数秒以内に集計基盤へ流したい」→ Kinesis Data Streams。

### Kinesis Data Streams vs Data Firehose

| 観点 | Kinesis Data Streams | Data Firehose |
|------|----------------------|---------------|
| **主目的** | ストリームを自前/柔軟に消費 | 宛先へほぼ自動配信 |
| **消費者** | アプリが読む | Firehose が配信 |
| **保持** | 長い | バッファ後に配送 |
| **向く用途** | 複数消費者、独自処理、再読 | S3/Redshift/OpenSearch への配送 |

#### サービス別の見分け方と出題例

- **Kinesis Data Streams**: シャード、コンシューマ、再読、複数アプリ処理、順序制御が出たら KDS。出題例: 「同じログストリームを不正検知とリアルタイム分析の 2 系統で読む」→ Kinesis Data Streams。
- **Data Firehose**: 配信先が S3/Redshift/OpenSearch/Splunk などで、運用を極力減らしたいなら Firehose。出題例: 「アプリログをほぼリアルタイムで S3 に蓄積し、変換後に OpenSearch へも届けたい」→ Data Firehose。

## 11. 可用性・DR 設計

### DR 4 戦略比較

| 戦略 | RTO | RPO | コスト | 概要 |
|------|-----|-----|--------|------|
| **バックアップ & リストア** | 高 | 高 | 最低 | 障害時に再構築 |
| **パイロットライト** | 中 | 低〜中 | 低 | コアのみ常時待機 |
| **ウォームスタンバイ** | 低 | 低 | 中 | 縮小構成を常時稼働 |
| **マルチサイト Active-Active** | 最低 | 最低 | 最高 | 複数環境を常時本番利用 |

#### 戦略別の見分け方と出題例

- **バックアップ & リストア**: コスト最優先で、復旧に時間がかかってもよいときの選択肢。出題例: 「災害はまれで、数時間〜数日の停止は許容」→ バックアップ & リストア。
- **パイロットライト**: データベースなど中核だけは生かしておき、アプリ層は災害時に起動する考え方。出題例: 「DB は常時複製したいが、アプリサーバーは平常時不要」→ パイロットライト。
- **ウォームスタンバイ**: ある程度動く縮小環境を常時持ち、障害時にスケールアップする。出題例: 「数分で切り替えたいが Active-Active ほどの予算はない」→ ウォームスタンバイ。
- **マルチサイト Active-Active**: ダウンタイムほぼ不可、両リージョンで常時処理したいならこれ。出題例: 「グローバルサービスでリージョン障害時も即時継続が必須」→ Active-Active。

### バックアップ最適化・自動化

| サービス | 対象 | 自動化 | 向く用途 |
|---------|------|--------|----------|
| **EBS Snapshot** | EBS | 手動中心 | 単発/基本操作 |
| **Data Lifecycle Manager** | EC2/EBS | スケジュール自動化 | EBS ライフサイクル管理 |
| **AWS Backup** | 複数サービス | ポリシー管理 | 組織横断バックアップ統制 |

#### サービス別の見分け方と出題例

- **EBS Snapshot**: 単発のポイントインタイム保存や、まず基本のバックアップを取りたい問題で使う。出題例: 「本番移行前に EBS ボリュームを一度だけ保全したい」→ EBS Snapshot。
- **Data Lifecycle Manager**: EC2/EBS 限定で、スナップショットの定期取得と世代管理を自動化したいとき向き。出題例: 「毎日深夜に EBS スナップショットを取り、30 日で削除したい」→ DLM。
- **AWS Backup**: RDS/EFS/DynamoDB/S3 など複数リソースをまとめてポリシー管理したいなら AWS Backup。出題例: 「複数アカウントのバックアップポリシーを統一したい」→ AWS Backup。

### Route 53 ルーティングポリシー

| ポリシー | 用途 | 特徴 |
|---------|------|------|
| **フェイルオーバー** | DR/HA | 主系障害時に待機系へ切替 |
| **加重（Weighted）** | 段階移行/A-B | 重み比率で分散 |
| **レイテンシ（Latency）** | 低遅延 | 最低レイテンシリージョンへ |
| **地理的（Geolocation）** | 地域別固定 | 国/地域で振り分け |
| **地理的近接性（Geoproximity）** | 地域範囲調整 | バイアスで範囲調整 |
| **複数値回答（Multivalue Answer）** | 簡易可用性向上 | 複数 IP を返す |

#### ポリシー別の見分け方と出題例

- **フェイルオーバー**: プライマリが落ちたらセカンダリへ自動切替、という文脈そのもの。出題例: 「東京リージョン障害時のみ大阪リージョンへ向けたい」→ フェイルオーバー。
- **Weighted**: 徐々に新環境へ流す、10% だけ新バージョンへ送る、という問題で使う。出題例: 「新 API へトラフィックの 5% だけ流して様子を見たい」→ Weighted。
- **Latency**: 地域ではなく実測レイテンシで最適先を選ぶ。出題例: 「世界中のユーザーを最も応答の速いリージョンへ送りたい」→ Latency。
- **Geolocation**: 国・地域・大陸など属性で固定的にルーティングしたいならこちら。出題例: 「EU ユーザーは必ず EU 内エンドポイントへ送る」→ Geolocation。
- **Geoproximity**: 地理的距離に加えてバイアス調整までしたい、という少し高度な設問向け。出題例: 「特定リージョンの集客範囲を意図的に広げたい」→ Geoproximity。
- **Multivalue Answer**: ELB ほどではないが、複数 IP とヘルスチェックで簡易分散したいときに使う。出題例: 「少数の Web サーバーへシンプルに分散し、死活監視もしたい」→ Multivalue Answer。

## 12. 直前確認用・頻出ひっかけメモ

- **NLB は `TCP/UDP/TLS` 系、ALB は `HTTP/HTTPS` 系**。問題文に `パスベース` があれば ALB、`固定 IP` や `UDP` があれば NLB を優先。
- **KMS は自動ローテーション可能**。ただし「DB パスワードの自動ローテーション」は Secrets Manager。
- **DataSync は常にエージェント必須ではない**。オンプレ接続ではよく使うが、AWS 間転送はエージェント不要ケースあり。
- **SNS にも FIFO Topic がある**。ただし「バッファリングして後で読む」は SQS。
- **IAM Identity Center は workforce SSO**。顧客向けログイン基盤は Cognito User Pool。
- **DocumentDB は MongoDB 互換**。完全な MongoDB と同一ではなく、「互換」「マネージド化」がヒント。
- **CloudWatch と CloudTrail と Config の問い分け**。
  - 今どうなっているか → CloudWatch
  - 誰が何をしたか → CloudTrail
  - 設定がどう変わったか/逸脱したか → AWS Config
